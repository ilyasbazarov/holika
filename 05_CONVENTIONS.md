# 05 · CONVENTIONS — Стандарты + правила агента

**Версия:** 1.0 · **Статус:** SEMI-STABLE
**Это самый важный документ для переносимости.** Он управляет поведением каждого рабочего чата.
Читается первым в любой сессии (вместе с `_METHOD`).

> **Состояние на P0:** Часть I (правила агента), Часть III (session-блоки), Часть IV (генерация брифов) —
> заполнены доменно-нейтрально из метода. **Часть II (нейминг + код-стандарт) — заглушка**, наполняется из
> фактов проекта в фазе P4 через ADR. Не выдумывать нейминг заранее.

---

## ЧАСТЬ I — ПРАВИЛА АГЕНТА (обязательны для разработчика)

### Роли
- **Архитектор** (дорогая модель) — проектирует, планирует, правит эту документацию, принимает решения
  (ADR). Прод-код задач не пишет. Бутстрап — `_ARCHITECT.md`.
- **Разработчик** (средняя модель) — берёт ОДИН task-brief, пишет код/артефакты. **Не исполняет** —
  исполняет человек.
- **Человек** — исполнитель и владелец решений. Запускает, возвращает логи, коммитит.
- **Applier** (дешёвая модель) — механически применяет `SESSION`-блок к артефактам репо. Бутстрап —
  `_APPLIER.md`.

### Модель исполнения (железно)
Разработчик **пишет** код/артефакты с ожидаемым выводом и инструкцией запуска. Человек **запускает** и
возвращает логи. Разработчик анализирует логи → следующий шаг или починка. **Задача НЕ выполнена без
подтверждённого человеком лога** — не по изолированной проверке, а по реальному интегрированному прогону.

### Ритуал СТАРТА сессии
1. Загрузить обязательный контекст: `_METHOD` + `00_CHARTER` + `05_CONVENTIONS` + `07_STATE` + task-brief +
   всё из его «context-to-load».
2. **Переформулировать задачу своими словами** и перечислить полученные входы/документы.
3. Проверить полноту. Нет чего-то из «context-to-load» → anti-improvisation.

### ★ Anti-improvisation (главный закон)
Требуемый документ/вход/значение/решение **отсутствует** в контексте → модель **ОСТАНАВЛИВАЕТСЯ**:
```
CONTEXT GAP: <что именно отсутствует и почему без него нельзя продолжить>
```
и запрашивает у человека. **Запрещено** выдумывать, «дотягивать», подставлять правдоподобные значения,
угадывать пороги/имена/схемы. Лучше остановка, чем тихая фабрикация.

> Гэпы в этом проекте — норма (источник неполон). Гэп → discovery-петля (`_METHOD.md` §11), не тупик и не
> догадка: классифицировать, закрыть дискавери или явно `DEFER` в `07_STATE`.

### ★ Untrusted-вывод терминала
Внутренне противоречивый/повреждённый лог (задвоенные команды, обрывки прошлых строк, рассинхрон тела и
статуса) считается НЕДОСТОВЕРНЫМ. Модель НЕ примиряет противоречие, НЕ выбирает «правдоподобную» половину,
НЕ достраивает — запрашивает чистый повторный лог в разделимой форме и решает по нему.

### ★ Не-идемпотентные операции
Обрыв/timeout во время не-идемпотентной операции (`create`, `deploy`, `run`) → **запрещён слепой retry**.
Сначала read-only диагностика (`describe` / `list`), затем доисполнение ТОЛЬКО недостающего хвоста.

### Дисциплина изменений документации
- STABLE/SEMI-STABLE доки (`00–02, 05, 08, 09`) — менять ТОЛЬКО через ADR в `06_DECISIONS_LOG`. Разработчик
  не правит их сам — предлагает в `NEW_DECISIONS`.
- `06` (DECISIONS) — только append, никогда не переписывать прошлые ADR (новое решение = новый ADR,
  помечающий старый `superseded`).
- `07` (STATE) — обновляется каждую сессию через `STATE_PATCH`.
- Доки исполнения (`03, 04`) — по мере исполнения через session-блок.

### Граница контракта — красная линия
Слой не лезет через границу контракта (`00_CHARTER` §главный принцип) чинить чужой слой. Дефект правится
**фикс-форвардом в нужный слой** (мерить там, где видно; чинить там, где причина).

## ЧАСТЬ III — ФОРМАТЫ SESSION-БЛОКОВ (генерится дословно)

В конце каждой сессии рабочий чат выдаёт ОДИН блок ниже. Applier/человек применяет механически.

```md
=== SESSION LOG · <date> · <task-id> ===

## SESSION_LOG  (→ append в /docs, опц. CHANGELOG)
- Задача: <T-id> — <название>
- Сделано: <маркеры>
- Команды/логи ключевые: <кратко>
- Отклонения от плана: <если были, иначе "нет">

## STATE_PATCH  (→ применить к 07_STATE.md)
- Задача <T-id>: <старый статус> → <новый статус>
- Текущий фокус: <что дальше>
- Новые открытые вопросы: <список или "нет">   ← GAP-реестр discovery-петли
- Блокеры: <список или "нет">
- updated_at: <date>

## NEW_DECISIONS  (→ append в 06_DECISIONS_LOG.md как ADR; "нет" если нет)
- ADR-0XX: <заголовок>
  Контекст: <зачем>
  Решение: <что>
  Последствия: <что меняется; при сдвиге значения — ВСЕ затронутые файл/§/строка>
  Статус: accepted | proposed (требует апрува)

## NEW_CONVENTIONS  (→ предложение правки 05; "нет" если нет)
- <предложение, помечено proposed — не применять без апрува>

=== END SESSION ===
```

**Правила применения:** `STATE_PATCH` применяется всегда. `NEW_DECISIONS` — только `accepted` (append, номер
ADR = следующий по факту). `NEW_CONVENTIONS` — по умолчанию `proposed`, не применять без явного апрува. Это
защищает источник истины от тихого дрейфа.

## ЧАСТЬ IV — ГЕНЕРАЦИЯ БРИФОВ (масштабирование без тяжёлых чатов)

**Принцип:** ни один чат не держит весь проект. Брифы НЕ пишутся в одном вечном архитекторском чате. Бриф —
repo-driven задача: свежий чат читает репо и выдаёт бриф следующей задачи. Механика — `_GENERATOR.md`.

- **Механический бриф** (задача расписана в `04_ROADMAP`) → свежий дешёвый чат через `_GENERATOR.md`.
- **Бриф с суждением** (нужны решения/декомпозиция) → свежий архитектор-чат: сначала закрывает открытые ADR,
  логирует в `06`, потом генерит бриф.
- **Discovery-бриф** (закрытие гэпа) → return-this = обогащение репо, не прод-код (`_METHOD.md` §11).

**«Архитектор» — роль, не чат.** Любой свежий архитектор-чат становится архитектором, загрузив
`_METHOD`+`00`+`04`+`06`+`07`. Решения → `06`, состояние → `07`. Старые чаты ретайрятся.
