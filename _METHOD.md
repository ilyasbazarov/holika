# _METHOD — Docs-driven multi-agent метод (доменно-нейтральный)

**Версия:** 1.0 · **Статус:** STABLE
**Провенанс:** дистилляция боевого метода из `ilyasbazarov/docs` @ SHA `cb33fb4a336863b4481b9dea4a093459c390b552`. Доменная специфика (каталог/ERP/облако) вычищена. §11 — расширение поверх baseline, помечено явно.
**Что это:** операционная система для ведения проекта силами моделей-агентов + человека, где знание живёт в плоском наборе `.md`, а не в контексте чата.

---

## 1. Философия (почему метод такой)

- **Ни один чат не держит весь проект.** Контекст растёт и деградирует — метод выносит знание в репо, а чаты берут ровно нужный срез.
- **Роль — не чат.** Свежий чат становится ролью, загрузив её обязательный контекст. Старые чаты ретайрятся, не живут вечно.
- **Знание переживает сессию, только если зафиксировано** структурным блоком и закоммичено. Незакрытая блоком сессия = потерянное знание.
- **Лучше остановка, чем тихая фабрикация.** Догадка, подставленная за факт, разваливает проект дороже, чем честный стоп.

---

## 2. Роли (модели разной стоимости как агенты)

| Роль | Модель | Делает | НЕ делает | Обязательный контекст |
|---|---|---|---|---|
| **Архитектор** | дорогая | проектирует, планирует, правит доки, принимает решения (ADR), держит целостность, пишет/ревьюит брифы | не пишет прод-код задач, не коммитит | `00`+`04`+`06`+`07` (+спеки) |
| **Разработчик** | средняя | берёт ОДИН бриф, пишет код/артефакты с ожидаемым выводом и инструкцией запуска | не исполняет сам, не правит STABLE-доки по своей инициативе | `00`+`05`+`07`+бриф |
| **Applier** | дешёвая | механически применяет session-блок к файлам репо | никакого творчества, не «улучшает» формулировки | `_APPLIER` + затрагиваемые файлы |
| **Человек** | — | запускает код, возвращает логи, коммитит, владеет решениями | — | — |

«Один файл — одна роль.» Подмножество ролей допустимо (напр. только архитектор+разработчик+человек), инвариант сохраняется.

---

## 3. Модель исполнения (железно)

Разработчик **пишет** код/артефакты с ожидаемым выводом и командой запуска. Человек **запускает** и возвращает логи. Разработчик анализирует лог → следующий шаг или починка. **Задача НЕ считается выполненной без подтверждённого человеком лога** — не по изолированной проверке, а по реальному интегрированному прогону.

---

## 4. Главный принцип: жёсткий контракт

Метод держится на существовании **жёсткого контракта** — красной линии, которую слои не пересекают. Контракт бывает в двух формах (проект может опираться на одну или обе):

**(a) Развязывающий интерфейс.** Жёсткий артефакт/схема между слоями. Всё выше (как он строится) и всё ниже (что из него строится) — независимы. Производителя можно менять, не трогая потребителя. Это причина, по которой проект **фазируется без переписывания**.

**(b) Референс-оракул приёмки.** Внешний источник истины (эталонные цифры/значения), с которым выход проекта обязан **биться по цифрам**. Расхождение закрывается правилом, override-ом или ADR — **не тихим примирением** «более правдоподобной» половины. Первый автопрогон сверяется с эталоном; каждое расхождение объяснено и закрыто.

**Закон-следствие:** слой не лезет через границу чинить чужой слой. Дефект правится **фикс-форвардом в нужный слой**, а не костылём на своей стороне.

## 5. Таксономия документов (плоский репо, нумерованные `.md`)

| Документ | Роль | Изменчивость |
|---|---|---|
| `00_CHARTER` | конституция: миссия, **главный принцип/граница контракта**, scope (in/out), критерии успеха, карта документов | STABLE |
| `01_ARCHITECTURE` | топология, последовательность прогона | STABLE |
| `02_*_CONTRACTS` | схемы граничных артефактов и таблиц | STABLE |
| `03_*_SPEC` | доменная логика ядра | SEMI-STABLE |
| `04_ROADMAP` | эпики → задачи с ID + приёмка | LIVING |
| `05_CONVENTIONS` | нейминг, код-стандарт, **правила агента + протокол сессий** | SEMI-STABLE |
| `06_DECISIONS_LOG` | ADR, **append-only** | LIVING (append) |
| `07_STATE` | текущий статус, открытые вопросы, контрольные цифры, «текущий фокус» | LIVING (каждая сессия) |
| `08_TASK_BRIEF_TEMPLATE` | шаблон пакета-передачи разработчику | STABLE |
| `09_GLOSSARY` | доменные термины | SEMI-STABLE |
| `10_*_PLAYBOOK` | грабли по типам ресурсов/операций | LIVING |

Каркас ролей: `_ARCHITECT.md`, `_APPLIER.md`, `_GENERATOR.md`, `briefs/<TASK>.md`.

**Правило `04_ROADMAP`:** детально расписан ТОЛЬКО текущий эпик; будущие — заголовки + scope, без выдуманных задач (иначе нарушение anti-improvisation).

**Инвариант-минимум контекста для ЛЮБОГО рабочего чата:** `00_CHARTER` + `05_CONVENTIONS` + `07_STATE` + конкретный бриф. Остальное — по списку «context-to-load» брифа.

---

## 6. Дисциплины (законы)

- **★ Anti-improvisation (главный закон).** Нет нужного дока/входа/значения/решения в контексте → модель ОСТАНАВЛИВАЕТСЯ:
  `CONTEXT GAP: <что отсутствует и почему без него нельзя продолжить>` — и запрашивает у человека. Запрещено выдумывать, «дотягивать», подставлять правдоподобные значения, угадывать пороги/имена/схемы.
- **Работа на фактах.** Поднять релевантный док ДО высказывания. Где факт не подтверждён доком — это гэп, а не догадка.
- **Дисциплина изменений.** STABLE/SEMI-STABLE доки меняются ТОЛЬКО через ADR; `06` — только append (новое решение = новый ADR, старый помечается superseded); `07` обновляется каждую сессию через `STATE_PATCH`; blast radius правки перечисляется полностью.
- **Untrusted-вывод.** Внутренне противоречивый/повреждённый лог (задвоенные команды, рассинхрон тела и статуса) — НЕДОСТОВЕРЕН. Не примирять, не выбирать «правдоподобную» половину — запросить чистый перезапуск в разделимой форме.
- **Не-идемпотентные операции.** Обрыв/timeout во время `create/deploy/run` → запрещён слепой retry. Сначала read-only диагностика (`describe/list`), затем доисполнение только недостающего хвоста.

---

## 7. Ритуалы сессии

**Старт:**
1. Загрузить обязательный контекст (§5) + бриф + всё из его «context-to-load».
2. Переформулировать задачу своими словами; перечислить полученные входы.
3. Проверить полноту → нет чего-то → `CONTEXT GAP`.

**Конец:** выдать ОДИН session-блок (§8). Без него сессия не закрыта, знание теряется.

---

## 8. Форматы session-блока (генерится дословно, применяется механически)

```md
=== SESSION LOG · <date> · <task-id> ===

## SESSION_LOG  (→ append в /docs, опц. CHANGELOG)
- Задача: <T-id> — <название>
- Сделано: <маркеры>
- Команды/логи ключевые: <кратко>
- Отклонения от плана: <если были, иначе "нет">

## STATE_PATCH  (→ применить к 07_STATE.md)
- Задача <T-id>: <старый статус> → <новый статус>
- Текущий фокус: <что дальше>
- Новые открытые вопросы: <список или "нет">
- Блокеры: <список или "нет">
- updated_at: <date>

## NEW_DECISIONS  (→ append в 06_DECISIONS_LOG.md; "нет" если нет)
- ADR-0XX: <заголовок>
  Контекст: <зачем> · Решение: <что> · Последствия: <что меняется>
  Статус: accepted | proposed (требует апрува)

## NEW_CONVENTIONS  (→ предложение правки 05; "нет" если нет)
- <предложение, помечено proposed — не применять без апрува>

=== END SESSION ===
```

**Правила применения:** `STATE_PATCH` применяется всегда. `NEW_DECISIONS` — только `accepted` (append, номер ADR = следующий по факту). `NEW_CONVENTIONS` — по умолчанию `proposed`, не применять без явного апрува. Патч ссылается на несуществующую структуру → «СТРУКТУРНОЕ НЕСООТВЕТСТВИЕ», стоп, не угадывать.

---

## 9. Генерация брифов (масштабирование без тяжёлых чатов)

**Принцип:** брифы не пишутся в одном вечном архитекторском чате. Бриф — repo-driven задача: свежий чат читает репо и выдаёт бриф следующей задачи.

- **Механический бриф** (задача уже расписана в `04_ROADMAP`) → свежий дешёвый чат через `_GENERATOR.md`.
- **Бриф с суждением** (нужны решения/декомпозиция) → свежий архитектор-чат: сначала закрывает открытые ADR, логирует в `06`, потом генерит.

**Процедура генератора:** прочитать `07_STATE` → «текущий фокус» → по `04_ROADMAP` собрать цель/зависимости/артефакт/приёмку → по `08` заполнить шаблон → выдать самодостаточный бриф (роль, context-to-load с литеральными URL, шаги, приёмка, return-this, требование session-блока). Есть нерешённые ADR у задачи с суждением → сначала закрыть или поднять человеку.

---

## 10. Механика доступа

- **Чтение:** raw-URL, **пиннить по commit-SHA** (`.../<SHA>/<file>`) — обход мутабельного CDN-кэша ветки. Query-string cache-busting не работает; SHA недоступен → запросить прямую вставку текста.
- **Запись обратно:** НЕ прямым коммитом модели. Только через session-блок → человек/applier коммитит. Это защищает источник истины от тихого дрейфа.

## 11. [РАСШИРЕНИЕ] Discovery-петля закрытия гэпов

> **Провенанс:** за пределами эталонного baseline. В эталоне есть зачаток — `CONTEXT GAP` (стоп-и-спроси). Здесь он формализован в рабочий цикл — для проектов, где **исходное знание заведомо неполно** и гэпы это норма, а не исключение. Без этой петли команда либо встаёт на каждом гэпе, либо (хуже) начинает гадать.

**Цикл (гэп → обогащение репо):**
1. **DETECT.** Любая роль ловит `CONTEXT GAP` — факт/решение/артефакт отсутствует.
2. **CLASSIFY.** Архитектор квалифицирует: блокер или нет; тип (недостающий факт / нерешённое решение / отсутствующий референс-артефакт).
3. **DISCOVER.** Ставится **discovery-бриф** — задача добычи знания: ручная выгрузка из внешнего источника (напр. эталонные цифры из ERP → `/reference`), расследование, замер, интервью с человеком.
4. **CAPTURE.** Находка обогащает репо: новый ADR (`06`), референс-артефакт (`/reference` + запись в `02_*_CONTRACTS`), или патч дока. Не в чат — в репо.
5. **RESUME.** Исходная задача разблокирована подтверждённым фактом.

**Правила петли:**
- Гэп **не дотягивается догадкой** — он либо закрыт дискавери, либо явно `DEFER` с фиксацией в `07_STATE`.
- **Реестр гэпов** живёт в `07_STATE` «Открытые вопросы»; закрытие → ADR или референс-артефакт (не молчаливое удаление).
- Discovery-бриф — обычный бриф по `08`, но его «return-this» = обогащение репо, а не прод-код.
- Внешне добытые эталонные значения (напр. выгрузка ERP) фиксируются как **референс-оракул** (§4b): последующая приёмка бьётся об них по цифрам.
