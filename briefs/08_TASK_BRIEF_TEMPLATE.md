# 08 · TASK_BRIEF_TEMPLATE — Пакет-передача для рабочего чата

**Версия:** 1.0 · **Статус:** STABLE
**Как пользоваться:** архитектор (или человек) заполняет шаблон под конкретную задачу и вставляет в новый
рабочий чат ВМЕСТЕ с указанными в «context-to-load» документами. «Context-to-load» всегда содержит
литеральный raw-URL для каждого пункта, не только имя файла. **Один бриф = одна задача = один чат.**

---

## ШАБЛОН (копировать и заполнить)

````md
# TASK BRIEF · T-<id>

## Роль
Ты — разработчик проекта. Сначала прочитай `_METHOD` + `05_CONVENTIONS` (правила агента), затем действуй.
Модель исполнения: ты ПИШЕШЬ код/артефакты, человек ЗАПУСКАЕТ и возвращает логи. Ты не исполняешь сам.

## Цель
<одно-два предложения: что должно появиться на выходе>

## Context-to-load (обязательно прочитать перед работой)
- `_METHOD`, `00_CHARTER`, `05_CONVENTIONS`, `07_STATE`  (всегда, с литеральными raw-URL)
- <конкретные доки/секции под задачу>
- <эталонные артефакты/цифры, если нужны>
Если чего-то из списка нет в контексте → выдай `CONTEXT GAP` и остановись.

## Входы
<какие файлы/таблицы/значения уже существуют и доступны>

## Шаги
1. <…>
2. <…>

## Критерии приёмки (Acceptance)
- <проверяемое условие 1>
- <сходимость с эталоном-оракулом, если задача его касается>

## Что вернуть человеку (Return-this)
- <код/артефакт + точная команда запуска + какой вывод прислать назад>

## Вне scope этой задачи
- <чего НЕ делать, чтобы не расползлось>

## В конце сессии
Выдай SESSION-блок по формату `05_CONVENTIONS` Часть III
(`SESSION_LOG` / `STATE_PATCH` / `NEW_DECISIONS` / `NEW_CONVENTIONS`).
````

---

## Вариант: discovery-бриф (обогащение репо)

Тот же шаблон, но задача — **добыть знание**, а не построить прод-артефакт (`_METHOD.md` §11):
- **Цель:** закрыть конкретный гэп из `07_STATE` «Открытые вопросы».
- **Return-this:** обогащение репо — черновой ADR / `/reference`-артефакт + запись в `02_*_CONTRACTS` /
  патч дока. Точно указать, куда что коммитится.
- **Acceptance:** знание зафиксировано и трассируемо к источнику; открытый вопрос помечен на закрытие.
- **Вне scope:** прод-код (пишется отдельной задачей уже поверх закрытого гэпа).

---

## Правила заполнения

- «Context-to-load» — всегда с **литеральным raw-URL** каждого пункта (пиннить по SHA при перечитывании).
- Не раздувать scope: явный раздел «Вне scope» обязателен.
- Приёмка — только **проверяемые** условия (лог / число / сходимость с эталоном), не «работает».
