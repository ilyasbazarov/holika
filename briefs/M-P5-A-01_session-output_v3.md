## 1. Ретро-ADR (для `06`, `proposed`) — БЕЗ ИЗМЕНЕНИЙ с v1

### ADR-010 · Конвертация мультивалютных сумм (minor-units × rate.value) — `proposed`
Канон: `minor_units ÷ 100 × rate` документа (`rate.value` или текущий курс) до суммирования в KGS; `÷100 без ×rate`
запрещён (корень TD-RECON-01, 90× недооценка USD); фикс 2026-06-25. Источник: PR-33 п.6, RB-38 §26.
Последствия: `02 §валюты:266` → ссылка ADR-010; `09 §конвертация валют` кросс-ссылка.

### ADR-011 · Фильтрация статей — только DELETE-after-MERGE (Ghost Records) — `proposed`
Выгружаются все платежи; системные статьи — `DELETE` после `MERGE`; инвариант: никогда `if/continue` при выгрузке.
Полный `EXCLUDE_EXPENSE_IDS` = Q-10. Источник: PR-27, PR-34 п.26.
Последствия: `03 §fact_payments:344` → ссылка ADR-011.

## 2. ★ ADR-012 · Прод-март расходов = `sq_marts_expenses` (fact_payments); TO-BE не строится — `proposed`

> Именно этот ADR ты просил: «узнать факт и закоммититься». Факт установлен из репо; для промоушена в
> `accepted` — одна команда живой перепроверки (ниже).

- **Контекст.** Требовалось установить, что реально питает страницу «Расходы» и совпадает ли это с ADR-006.
  Репо противоречиво: ADR-009 говорит «Epic-1 **строит** TO-BE raw-март + ретайрит легаси», владелец — «проект
  уже построен, ничего заново не делаем». Разрешено фактом (см. шапку): прод-цепочка =
  `msklad_expenses`(LS pass-through) → `marts.expenses`(таблица) → `sq_marts_expenses`(SQ, читает
  `fact_payments` = paymentout+cashout). ADR-006 двухисточниковая методология (paymentout+cashout+**loss** +
  **commissionreportin**) в проде **НЕ реализована** — валидирована лишь ручной выгрузкой 2026-07-06.
- **Решение.**
  1. **Факт (канон):** прод-витрина расходов = `marts.expenses`, строится `sq_marts_expenses`
     (Config `6a22a243-0000-20fd-a458-883d24f4cad4`) из `core.fact_payments`. LS `msklad_expenses` —
     date-filtered pass-through, без агрегации/исключений/FX.
  2. **TO-BE raw-март из ADR-006/009 НЕ строится** (решение владельца). Мы **правим существующий**
     `sq_marts_expenses`, а не заменяем новым мартом.
  3. **Переформулировка курса (supersedes ADR-009 §1):** Epic-1 = **верификация** `marts.expenses` против
     эталона + **точечные правки** `sq_marts_expenses` там, где расхождение подтверждено (в первую очередь —
     добавить недостающие источники `loss` + `commissionreportin` по ADR-006). Не постройка с нуля.
  4. **Гипотеза расхождения (grounded):** прод-март структурно занижает статьи, питаемые из `loss`, и
     **«Расходы маркетплейсов»** (источник #2 `commissionreportin` отсутствует целиком). Это механизм
     задокументированного ~33% разрыва (ADR-005 §4). Подтверждается/опровергается в E1-T1.
  5. **Провенанс + верификация:** `/reference/sql/sq_marts_expenses.sql` @ снапшот 2026-07-07
     (SHA `805450a`, M-P4-D5). Живая перепроверка для промоушена в `accepted`:
     ```bash
     bq show --transfer_config --project_id=msklad-bi-prod \
       projects/msklad-bi-prod/locations/asia-east1/transferConfigs/6a22a243-0000-20fd-a458-883d24f4cad4
     # убедиться, что FROM всё ещё `core.fact_payments` и расписание не задано
     bq show --format=prettyjson msklad-bi-prod:marts.expenses   # + свежесть последнего прогона
     ```
- **Последствия.** `ADR-009` §1 (build-framing) → пометить `partially superseded by ADR-012`. `04_ROADMAP`
  Epic-1 → «верификация+правки», не постройка. `07` Q-18 → CLOSED (факт). `03 §marts.expenses` — при
  наполнении прозой отразить: прод = fact_payments-март, ADR-006 = целевая методология правки.
- **Статус:** proposed (промоушен в `accepted` — после `bq show`-перепроверки владельцем).
