## Шаги
1. **Заземлить трансформ (не предполагать).** Прочитай `/reference/sql/sq_marts_expenses.sql`. Явно выпиши:
   `FROM` (ожидание — `core.fact_payments`), список агрегируемых источников, ключ группировки, единицу суммы
   (`sum_kgs`), формулу USD. Зафиксируй: **`loss` и `commissionreportin` в трансформе отсутствуют структурно** или нет.
2. **Вывести структурный прогноз гипотезы (ADR-012 §4).** Из SQL — какие из 27 категорий *не могут* получить вклад
   `loss` / `commissionreportin`. Пометь как **ГИПОТЕЗУ к проверке цифрами**, не как вывод.
3. **Снять прод-факт за май-2026.** Напиши read-only `SELECT` по `marts.expenses` (или, если витрина без периода —
   по `sq_marts_expenses`-логике за май), агрегат **в KGS** по `expense_item_name`. Человек запускает, шлёт вывод.
   ```sql
   -- ЧЕЛОВЕК ЗАПУСКАЕТ. Read-only. Подставь фактические имена колонок из шага 1.
   SELECT expense_item_name, SUM(sum_kgs) AS prod_sum_kgs
   FROM `msklad-bi-prod.marts.expenses`
   WHERE DATE(moment) BETWEEN '2026-05-01' AND '2026-05-31'   -- сверь границу периода с оракулом
   GROUP BY expense_item_name
   ORDER BY prod_sum_kgs DESC;
   ```
3a. **Свежесть/провенанс прогона** (чтобы сверять актуальный снимок):
   ```sql
   bq show --format=prettyjson msklad-bi-prod:marts.expenses   -- последний прогон / lastModifiedTime
   ```
4. **Построить построчную сверку (в KGS, единица оракула).** Таблица: `категория (27)` · `prod_sum_kgs` ·
   `oracle_kgs` (ORACLE-строка из `pnl_2026-05.md`) · `diff_abs` · `diff_%`. Матч по **имени** статьи (регистр
   критичен). USD — вторичен/derived, не основа сверки.
5. **Классифицировать каждую строку:** `MATCH` (до копейки) · `UNDERSTATED` · `OVERSTATED` · `MISSING`. Для
   `UNDERSTATED`/`MISSING` — пометь, питается ли статья из `loss` или это «Расходы маркетплейсов» (источник #2
   `commissionreportin`). **Сопоставь картину с прогнозом шага 2 → гипотеза ADR-012 §4 ПОДТВЕРЖДЕНА или ОПРОВЕРГНУТА.**
6. **Валютная развязка (не смешивать классы багов).** Проверь: `total_sum_usd` в проде = единый последний курс на
   всю историю (ADR-012). Если это даёт отдельное расхождение — это **другой класс**, чем loss/commission-гэп; веди
   отдельной строкой. Класс `×rate`-регрессии (TD-RECON-01 / Q-20) — вне scope, но если увидел — зафиксируй как находку.
7. **Черновой диагностический ADR на каждое подтверждённое расхождение** (причина, трассируемо к оракул-строке +
   SQL-снапшоту). Правку НЕ писать — это отдельная задача.
8. **Граница диагностики (Q-3).** Если корневая причина требует *неверсионированного* исходного кода пайплайна
   (напр. конформность DELETE-after-MERGE по ADR-011, полный `EXCLUDE_EXPENSE_IDS` = Q-10) — **не гадать**: выдай
   `CONTEXT GAP → DEFER Q-3` и зафиксируй в GAP-реестре. Снапшот SQL покрывает только трансформ марта, не весь пайплайн.
