## Известные факты-грунт (не пере-открывать — использовать)
- **Живой март = чистый SQL** `SUM(p.sum_kgs) FROM core.fact_payments` (2 из 4 источников: только `paymentout`+`cashout`; нет `loss`, нет `commissionreportin`) + `LEFT JOIN core.dim_fx_rates` для исходящего USD-курса. cf-finance он НЕ вызывает — он читает его выходную таблицу.
- **Колонка `p.sum_kgs` несёт валютный баг ADR-016** (`÷100` без `×rate.value` для не-KGS): не-KGS `paymentout`-строки в проде **системно занижены** (38/38 не-KGS докум. мая, ≥7 статей, Σ 1 108 075,75 KGS). Это дефект слоя B (cf-finance) — **чинится отдельной задачей `E1-T3-MECH-FX`**, НЕ здесь (закон-следствие: фикс-форвард в нужный слой, `00_CHARTER`).
- **Все расходные цифры дашборда заморожены до `E1-T3-MECH-FX`** (ADR-016 §Последствия) — не предъявлять как финальные.
- **Регистр-миссмэтч** (фильтр по имени, ADR-006, case-critical): клиентские `ТОпливо`, `неразнесенные списания` vs прод `Топливо`, `Неразнесенное списание` — потенциальный тихий промах при case-sensitive сравнении.
- «Вывод прибыли» — движение капитала (ADR-013 §1), **вне** 26 опер-категорий. «Налоги и сборы» — начисление П&Л, 2-й источник, вне cash exact-match (ADR-013 §3) — форма → `E1-T2-MECH-TAX`, не здесь.

## Шаги

1. **Закрыть Q-31 (grounding, ПЕРВЫМ; не фабриковать).**
   Прочитай `/reference/sql/sq_marts_expenses.sql` и `03 §marts.expenses`. Реши развилку **фактом + дизайном**, откуда правка возьмёт `loss`/`commissionreportin`:
   - **α** — из cf-finance-таблицы `core.fact_*` → баг ADR-016 наследуется → **cutover E1-T1-MECH обязан идти ПОСЛЕ `E1-T3-MECH-FX`**.
   - **β** — напрямую из `raw.moysklad_loss` / `raw.moysklad_commissionreportin` c FX по ADR-010 (`÷100 × rate.value`) в самом SQL → новые источники баг не наследуют; но пред-существующие `paymentout`/`cashout` всё ещё тянут баг из `core.fact_payments` → **смешанное состояние** (флагни явно).
   Выдай вердикт как `NEW_DECISIONS` (`proposed`, требует апрува владельца/архитектора): выбранная ветка + следствие для порядка cutover. НЕ выбирай α/β «по правдоподобию» — обоснуй тем, где `raw.moysklad_loss`/`commissionreportin` реально живут (проверяемо `bq ls`/`bq show`, если человек запустит) и что дешевле/корректнее. При недостатке факта для выбора → `CONTEXT GAP`, а не догадка.

2. **Снять прод-срез.** Напиши SQL, дающий `marts.expenses` за **май-2026** в разрезе `expense_item_name` (Σ `total_sum_kgs`, `payment_count`). Ожидаемый вывод — таблица «статья → сумма KGS».

3. **Построчная сверка vs оракул.** Сопоставь прод-срез с 26 дашборд-категориями `/reference/pnl_2026-05.md`. Нормализуй имена (case-insensitive matching) — но **пометь** каждый регистр-миссмэтч отдельно, не «выравнивай» молча.

4. **Атрибуция расхождений (ядро задачи).** Для каждой не-нулевой дельты классифицируй причину, **не смешивая три ортогональных дефекта**:
   - `MISSING_SOURCE` — занижение из-за отсутствия `loss` и/или `commissionreportin` (гипотеза ADR-012 §4; ожидается на статьях с loss-компонентой и на «Расходы маркетплейсов» без `commissionreportin`);
   - `FX_BUG` — занижение не-KGS строк из-за бага cf-finance (ADR-016) — **это не чинится в E1-T1**, только атрибутируется (иначе двойной учёт причины);
   - `NAME_CASE` — промах фильтра по регистру имени;
   - `OTHER` — всё прочее, с гипотезой.
   Результат — таблица `статья | прод_KGS | оракул_KGS | дельта | причина | примечание`.

5. **Дизайн точечной правки (не исполнять).** Опиши SQL-дельту к `sq_marts_expenses`, добавляющую недостающие источники по выбранной в Шаге 1 ветке, с FX по ADR-010 и фильтром по имени (ADR-006, с учётом регистра). Требование `E1-T1-MECH` / ADR-012: **staging-first** — правка проверяется на staging/копии, прод `transferConfig` `6a22a243-…` (живой ежедневный деструктивный WRITE_TRUNCATE) in-place НЕ трогается в этой задаче.
