## Шаги

### Шаг A. Старт-ритуал + сверка состояния (СНИМАЕТ причину прошлого стопа)
1. Переформулируй задачу своими словами, перечисли полученные входы/доки (`05` §Ритуал СТАРТА).
2. **Процитируй ДОСЛОВНО** из только что прочитанного `07_STATE` @ `1dd7cde`: секцию «Текущий фокус» и
   строку статуса задачи `E1-T3-MECH-FX`.
3. **Сверка-гейт (STOP-условие):** этот бриф утверждает, что @ `1dd7cde` Шаги 2/3/5 закрыты по факту, а
   **единственный незакрытый gate — Шаг 4 (staging dry-compare), артефакт которого в репо отсутствует.**
   - Если процитированный `07_STATE` этому НЕ противоречит (RQ-2/Шаг 3 могут быть помечены как незакрытые —
     это ожидаемо, их закрывает Шаг B ниже фактом с диска) → продолжай.
   - Если `07_STATE` показывает, что Шаг 4 УЖЕ закрыт с зафиксированным `diff = 0,00` артефактом → **STOP**,
     сообщи человеку: «Шаг 4 уже закрыт, брифу нечего проверять, нужен ли сразу cutover?». Не дублируй.
   - Если приложенный SHA не `1dd7cde` и содержимое разошлось → **STOP**, запроси актуальный SHA
     (`_START_HERE ★★`: работать на устаревшем контенте запрещено).

### Шаг B. Верификация фикса на диске = финальная ревизия (read-only)
Напиши read-only bash-блок (человек запустит), который:
1. `sha256sum /home/ilyasbazarov4/cf-finance/main.py` — зафиксировать хэш ревизии, что идёт в cutover.
2. Выводит строку конверсии: `grep -n 'sum_kgs' /home/ilyasbazarov4/cf-finance/main.py`.
3. Подтверждает, что формула содержит `× rate.value`-множитель (иначе фикс на диске НЕ финальный) и
   что `main.py.pre-e1t3-mech-fx.bak` существует (откат возможен).
- **STOP-условие:** если `× rate.value` в строке `sum_kgs` отсутствует → фикс не финальный/потерян →
  вынеси человеку, не изобретай фикс заново (он адъюдицирован, форма известна — восстанови из ADR-010).
- Это закрывает RQ-2 по факту: форма `row["rate"]["value"]` в рабочем коде = API отдаёт `rate` вложенным
  объектом. Зафиксируй в Return-this как факт, а не догадку.

### Шаг C. Шаг 4 — staging dry-compare (форма (б), БЕЗ записи в прод) — ГЛАВНЫЙ GATE
Форма зафиксирована архитектором как **(б) dry-compare** (не staging-MERGE). Обоснование — в разделе
«Почему (б)» ниже; НЕ переигрывать на MERGE-вариант «для реализма».

Напиши Python-скрипт (`/tmp/e1t3_drycompare.py` или аналог, human-run), который:
1. Тянет из **живого API МойСклад** документы `paymentout` + `cashout` за **май-2026** (тот же пагинат-обход
   и фильтр периода, что в `main.py`; переиспользуй его код чтения — не переписывай логику запроса заново).
2. Для каждого документа считает `sum_kgs` **ровно той формулой, что лежит на диске `main.py:69`** —
   импортируй/скопируй строку буквально, не набирай по памяти (иначе проверишь не тот код, что поедет в прод).
3. Отдельно берёт **не-KGS** документы (валюта ≠ KGS, т.е. `rate.value` присутствует и ≠ 1) и сверяет
   исправленный `sum_kgs` против эталона — суммы, выведенной из `rate.value` **того же ответа API**.
4. Печатает: число не-KGS документов; число KGS-документов; **суммарный и построчный diff (исправленный
   sum_kgs − эталон) по не-KGS**; отдельно — что KGS-документы формула НЕ изменила.
- **Цель gate: diff по не-KGS май = 0,00 БЕЗ ОСТАТКА; KGS-документы неизменны.**
- **STOP-условие (зелёный):** diff = 0,00 → **ОСТАНОВИСЬ. Верни лог человеку. НЕ переходи к Шагу D.**
  Апрув на cutover — отдельное явное слово человека, не следствие зелёного лога.
- **STOP-условие (красный):** diff ≠ 0,00 → **ОСТАНОВИСЬ. Диагностика, НЕ подгонка.** Вынеси архитектору:
  что за документы дают остаток, какая валюта, какой `rate.value`. Не правь формулу «на глаз» до 0,00.

### === STOP-ГЕЙТ (между Шагом C и Шагом D) ===
**Шаг D НЕ начинается автоматически после зелёного Шага C.**
Разработчик обязан дождаться, пока человек в ответ на зелёный лог Шага C напишет **явно**: `апрув cutover`
(или равнозначно недвусмысленно). Пока этого слова нет — сессия стоит на STOP. Это первый деструктивный шаг
в проде; апрув не выводится из данных и не презюмируется (`05` Часть I — владелец решений; ADR-019 §4 порядок).

### Шаг D. Шаг 6 — контролируемый прод-cutover (ТОЛЬКО после `апрув cutover`)
Напиши единый cutover-скрипт (`ADR-014`: один copy-paste блок, self-resolving, ожидаемый вывод на каждом
этапе), последовательность:
1. **pause** `finance-daily-update` (Cloud Scheduler) — остановить расписание.
2. **describe/list no in-flight** — убедиться, что нет активного прогона cf-finance (иначе гонка с deploy).
3. **deploy** `main.py` (ревизия с зафиксированным в Шаге B sha256) в `cf-finance`.
4. **manual run** — один ручной прогон обновлённой функции.
5. **re-diff прод** `core.fact_payments` vs живой API по не-KGS май = **0,00 без остатка** (боевой аналог
   Шага C, теперь на реальном проде).
   - **STOP-условие:** re-diff ≠ 0,00 → НЕ resume. Откат на `main.py.pre-e1t3-mech-fx.bak`, вынести
     архитектору. Прод не оставлять в промежуточном состоянии.
6. **resume** `finance-daily-update` — вернуть расписание.
- Каждый этап печатает ожидаемый маркер; человек возвращает полный лог.

### Шаг E. Пост-cutover (обогащение провенанса)
Снапшот новой ревизии `main.py` + sha256 → куда указано в Return-this. Зафиксировать факт deploy.

  не-KGS + подтверждение неизменности KGS [Шаг C]; (3) [после апрува] прод re-diff = 0,00 [Шаг D];
  (4) подтверждение pause/resume; (5) снапшот новой ревизии + sha256 [Шаг E].

  не патч-цепочка на собственные неприменённые патчи). В `STATE_PATCH`:
  - `E1-T3-MECH-FX` → **DONE** только при зелёном прод re-diff [Шаг D]; если сессия остановилась на STOP-гейте
    (Шаг C зелёный, апрува нет) → статус «Шаг 4 PASS, cutover ждёт апрува», НЕ DONE.
  - RQ-2 → CLOSED (факт с диска, Шаг B).
  - Текущий фокус → следующее по `04_ROADMAP`; при зелёной приёмке — снять заморозку расходных цифр
    дашборда (ADR-016 §Последствия); коснуться Q-31 (порядок cutover E1-T1-MECH), если маршрутизация
    подтверждена.
  - **STATE_PATCH фиксирует статус только при наличии артефакта возврата.** «DONE» без возвращённого лога
    прогона — claim, не факт (урок этой цепочки: строка `07_STATE` разошлась с диском именно так).

## Почему форма (б) dry-compare, а не (а) staging-MERGE (решение архитектора — не переигрывать)
Проверяется **одна изменённая строка арифметики** (`× rate.value`), не пайплайн — пайплайн уже в проде.
(б) мерит исправленную формулу напрямую против `rate.value` живого API из того же ответа — прямой замер,
минимум движущихся частей между формулой и эталоном. (а) прогнал бы весь конвертер + MERGE в копию, добавив
шум чтения/записи/MERGE между тобой и ответом: при расхождении не отличить «формула кривая» от «обвес кривой».
MERGE-механику мы не меняем (Q-19: конвертер локальный, inline, blast radius ограничен) → проверять её в
gate незачем. Интеграционный тест на реальном проде у нас и так есть — это re-diff Шага D. (б) на gate +
re-diff на cutover покрывают арифметику и интеграцию без нахлёста.
