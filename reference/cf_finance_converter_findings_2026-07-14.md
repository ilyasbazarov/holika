## 1. Локус конвертера (заземление Q-27 / ADR-016 §3)

**Файл:** `main.py`
**Функция:** `run_etl()`
**Строка (в снапшоте — 67-я строка тела функции, участок формирования `records`):**

```python
"sum_kgs": float((row.get("sum") or 0) / 100.0),
```

**Формула as-is:** `sum_kgs = sum ÷ 100`. **Ни разу** в `main.py` не встречается идентификатор `rate` (grep по `rate\.value|rate\[` на всём файле дал 0 совпадений вне этой находки — см. `cf_extract.log` шаг 5, все совпадения по паттерну относятся только к `sum_kgs`/`paymentout`/`cashout`, ни одного к `rate`). Умножение на курс документа (`rate.value`, канон ADR-010) в коде **отсутствует полностью** — не «забыто в одном месте», а не реализовано как концепция: переменная/поле `rate` нигде не читается из ответа API и не участвует в расчёте.

**Дополнительное системное наблюдение (не было explicit в брифе, но релевантно root cause):** HTTP-запрос к МойСклад API строится с `expand=expenseItem,agent,project,salesChannel` (строка `url = f"https://api.moysklad.ru/...&limit=100"`) — `rate` **не входит** в `expand`. Установить без дополнительного discovery, приходит ли `rate` в ответе МойСклад по умолчанию (без expand) или требует явного expand — **CONTEXT GAP**, не подтверждено в этой сессии (нет доступа к сырому API-ответу с текущим кодом; это вопрос API-контракта МойСклад, не тела `main.py`). Не блокирует находку локуса (код в любом случае не использует `rate` даже если бы он приходил), но релевантно для `E1-T3-MECH-FX`: фикс может потребовать правки `expand`-параметра запроса, а не только формулы — оставляю как заметку для брифа фикса, не решаю здесь.

**Формула ADR-010 (канон, для сравнения):** `minor_units ÷ 100 × rate.value` документа (если задан; иначе текущий курс валюты). Код `main.py` реализует только первый множитель, второй отсутствует безусловно — для ЛЮБОЙ валюты документа, не только не-KGS. Для KGS-документов `rate.value` в API обычно не задан/=1, поэтому баг не проявляется числово (эквивалентно `×1`) — это ровно наблюдение из `recon_vyvod_pribyli_2026-05.md §2` (23/25 KGS-документов совпали, 2/25 не-KGS разошлись).

## 2. Структурный охват (Q-30/Q-19 структурная половина — БЕЗ квантификации)

**Импорты `main.py` и `patch_main_finance.py`** (`cf_extract.log` шаг 6):

```
main.py:
  import os
  import time
  import requests
  import datetime
  from google.cloud import bigquery
  from google.cloud import bigquery_datatransfer
  from google.protobuf.timestamp_pb2 import Timestamp

patch_main_finance.py:
  import shutil
  import sys
```

**Вывод: конвертер — ЛОКАЛЬНЫЙ (inline), не общий модуль.** Ни одного импорта из общего/shared/lib/utils-модуля не найдено (эвристика по `^from (common|shared|lib|utils)` — 0 совпадений; полный список импортов выше исчерпывающий — только stdlib + google-cloud SDK). Строка `sum_kgs = sum / 100.0` — единственное вхождение расчёта суммы, определено непосредственно в теле `run_etl()`, не вызывает внешнюю функцию-конвертер.

**Практическое следствие для блast radius (Q-30/Q-19):**
- Баг `÷100 без ×rate` в этом коде **не распространяется автоматически** на другие `core.fact_*` через общий код-путь — потому что общего кода-пути нет, конвертер cf-finance не переиспользуется.
- Это **закрывает структурную половину Q-3-расширения**, но **НЕ закрывает** Q-19 (sales-сторона, `sq_marts_sales_overview`) и Q-30 (магнитуда по другим таблицам) — те таблицы (`fact_sales_profit` через `cf-facts`, и т.д.) питаются **другими Cloud Functions** с собственным, отдельно не проверенным кодом. Если у `cf-facts`/других CF есть **свой независимый** `÷100`-без-`×rate` баг — это тот же класс дефекта, но НЕ тот же код и НЕ доказывается этой находкой. Требует отдельного discovery по каждой CF (вне scope этой задачи, см. «Вне scope»).
- Формулировка для `07_STATE`: конвертер `cf-finance` **локален для cf-finance**; структурная гипотеза «общий модуль, питающий весь `core.fact_*`» — **опровергнута фактом** (импорт-граф пуст). Blast radius бага, доказанного в `recon_vyvod_pribyli_2026-05.md`, ограничен таблицами, которые загружает именно `cf-finance` — то есть `core.fact_payments` (`paymentout`+`cashout`) и, транзитивно, всё, что читает `core.fact_payments` (`marts.expenses` через `sq_marts_expenses`, потенциально другие SQ, не проверялись). Не расширяется на `fact_sales_profit`, `fact_purchases`, `fact_returns`, `fact_inventory` — те не проходят через этот код.

## 3. Provenance / disk-vs-deployed (сводка, детали — `MANIFEST.md`)

Сверка выполнена: код (`main.py`, `patch_main_finance.py`, `requirements.txt`) байт-в-байт совпадает между on-disk и задеплоенной ревизией `cf-finance-00006-piv`. Единственное расхождение — `.gcloudignore` присутствует только на диске (build-control файл, не код). Находка локуса конвертера **трассируема напрямую к продовому поведению** — не к устаревшей/локальной версии кода.

## 4. Что НЕ измерено этой сессией (явные границы)

- Магнитуда бага по `cashout`, другим месяцам, всему марту — `Q-30`, DEFER, вне scope.
- Наличие/отсутствие аналогичного бага в `cf-facts`/`cf-fx`/`cf-dim`/`cf-inventory` — не проверялось, эти CF не извлекались (residual `Q-3`, следующий discovery-бриф).
- Приходит ли поле `rate` от МойСклад API без явного `expand=rate` — открытый вопрос API-контракта (см. §1), не решён.
