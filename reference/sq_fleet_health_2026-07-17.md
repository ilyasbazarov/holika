Дельта = ровно одна колонка. `02 §схемы dim` (14 колонок) согласован с живой `core.dim_products` — расходится цель.

**Следствие для формы фикса:** `ALTER TABLE … ADD COLUMN weight` кладёт `weight` на поз.15, тогда как `snapshot_at` остаётся на поз.14; `INSERT … SELECT *` матчит **по позиции** ⇒ FLOAT64 → TIMESTAMP, падение сохраняется. Голый ALTER не чинит (ADR-019 §3).

## Ф-2. Хронология (Q-33а, Q-36)

| Дата | Событие |
|---|---|
| 2026-05-07 13:19:53 … 13:27:29 | заводка/правка конфига (= `updateTime` из `/reference/sql/README` §Наблюдения п.2) + **3 ручных прогона** (13 389 строк = ровно 3 × 4 463), вне расписания и вне run history ⇒ выполнены не через DTS; инструмент не установлен |
| 2026-05-08 04:00:12 | первый прогон **по расписанию**, SUCCEEDED |
| 2026-06-02 04:00:10 | **последний успешный** (`MAX(snapshot_at)`) |
| 2026-06-03 | **первое падение** — schema drift |
| 2026-07-17 | состояние на дату снятия: FAILED, 45 суток подряд |

`updateTime` 2026-05-07 предшествует первому прогону по расписанию ⇒ **не является датой падения**. `/reference/sql/README` §Наблюдения п.2 не опровергнут, но склеивал три разных события в одно.

**Подтверждение дыры тремя независимыми источниками:**
1. `MAX(snapshot_at)` = 2026-06-02 04:00:10;
2. run history — 71 прогон, первый 2026-05-08; 05-08…07-17 = 24+30+17 = 71 ⇒ покрытие сплошное, суточное, без пропусков;
3. последние 50 прогонов = 5 SUCCEEDED (05-29…06-02) + **45 FAILED** (06-03…07-17).

⇒ дыра = 2026-06-03 … 2026-07-17 = **45 суток**. `dim_products` в ядре без SCD2 (перезапись MERGE, `ADR-008 §Контекст`), снапшот = единственный носитель истории (`ADR-008 §3`) ⇒ срезы за окно **безвозвратны**, задним числом не доливаются.

## Ф-3. Объём (закрывает `ADR-008 §4` количественно)

`rows_total` = 129 905 · `distinct_snapshot_days` = 27 (26 по расписанию + 1 ad-hoc). Суточный срез 4 463 → 4 492 ⇒ **~4 492 строк/сут, ~1,64 млн строк/год**, порядок сотен МБ. `DEFER` по retention подтверждён цифрой, не умолчанием.

**Контрольная сверка:** 13 389 + 4 463×6 + 4 471 + 4 475×4 + 4 491×13 + 4 492×2 = **129 905** = `rows_total` ровно ⇒ иных вставок таблица не содержит.

## Ф-4. Латентный дефект актива (Q-37)

По `DATE(snapshot_at) = 2026-05-07` лежат **три** среза вместо одного. Наивный `SELECT … WHERE DATE(snapshot_at) = '2026-05-07'` утроит строки. Сегодня безвреден (нет потребителя, `ADR-008 §2`); срабатывает при промоушене таблицы в consumer-facing роль.

## Ф-5. Лид на причину дрейфа — НЕ установлено

В `02 §схемы dim` блок, содержащий `weight` в `core.dim_products`, закрыт маркером `(PR-26)`; в том же блоке `core.dim_fx_rates` несёт «Источник **с 2026-06-03**». Окно дрейфа, установленное независимо, — между 2026-06-02 04:00 и 2026-06-03 04:00 UTC. Совпадение точное; `PR-26` встречается во всём наборе доков ровно один раз (это PR живого код-репо, `ADR-017`, не ADR).

**Не утверждается** (маркер накрывает блок, дата привязана к строке про курсы). Закрывается взглядом в `PR-26`. Практической нужды нет: ADR-019 §3 (вариант E) делает подобные изменения безопасными независимо от того, помнил ли их автор про audit-снапшот.
