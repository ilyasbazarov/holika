## 3. Δ(B↔C) — граница месяца / бакетинг

Все 3 предиката бакетинга (`year_month='2026-05'`, `month_start='2026-05-01'`, `DATE(moment) BETWEEN '2026-05-01' AND '2026-05-31'`) на `marts.expenses` дали идентичный результат прод-цифре B (3 355 976,16 / 25 док). **Δ(B↔C) = 0,00.**

**Побочный факт, обнаруженный при диагностике границы:** запрос сравнения `moment` с `TIMESTAMP`-литералом упал (`No matching signature: DATE, TIMESTAMP`). Это подтверждает: **`core.fact_payments.moment` — тип `DATE`**, не `TIMESTAMP` (колонка не была задокументирована в `02_ERP_CONTRACTS §схемы core` — та же категория гэпа, что Q-4). Так как `moment` уже `DATE` (без времени суток), TZ-граница UTC↔Asia/Bishkek физически не может двигать документ через границу месяца на уровне BQ-хранения этой колонки. В сочетании с полным совпадением id-сета (25=25 между A и B) — потери документов на границе мая **не было**.

## 4. Инвариант декомпозиции

**Δ(A↔B) + Δ(B↔C) = 890 187,50 + 0,00 = 890 187,50** — точное совпадение с исходным разрывом Q-27. Необъяснённого остатка нет.

## 5. Staleness

- `MAX(_loaded_at)` для майских ВП-строк в `core.fact_payments` = `2026-07-13 21:10:35`.
- Последний успешный прогон `sq_marts_expenses` (`lastModifiedTime` таблицы `marts.expenses`) = `2026-07-13T11:10:01Z` (~10ч раньше).
- **Наблюдение (не блокер, не источник разрыва):** прогон марта хронологически предшествует последней перезагрузке этих строк в core в тот же день, но при этом B=C совпадают точно — то есть перезагрузка `core` после 11:10 не изменила значения ВП-строк (вероятно, повторная выгрузка тех же неизменившихся в МойСкладе документов, обновляющая только `_loaded_at`). Staleness **не является причиной** разрыва 890 187,50 (доказано отсутствием расхождения B↔C).

## 6. Ghost-DELETE-after-MERGE (`03 §fact_payments`)

Не проверено исчерпывающе — полный список `EXCLUDE_EXPENSE_IDS` канон в невыгруженном коде `cf-finance` (**GAP Q-10**, недоступен из репо). Но: doc-count B=25=A (полное совпадение id-сета) исключает возможность, что «Вывод прибыли» целиком или частично вымывается DELETE-правилом — если бы это происходило, count в B был бы меньше count в A. **Ghost-DELETE как причина Q-27 отклонён фактом** (не строк не хватает — цифры хватающихся строк неверны).

## 7. Root cause (атрибуция, доказательно)

**Механизм:** этап загрузки `paymentout`/`cashout` в `core.fact_payments` (предположительно Cloud Function `cf-finance`, `11_INFRA_FACTS §CF`) вычисляет `sum_kgs = sum ÷ 100`, но **не умножает на `rate.value` документа**, когда валюта документа ≠ KGS. Для документов в валюте KGS (`rate.value` отсутствует) это не проявляется. Для 2 документов «Вывод прибыли» в валюте USD и RUB — проявляется полностью и объясняет 100% разрыва без остатка.

Это ровно тот паттерн, который уже был предвиден **ADR-010** (§валюты, PR-33 правило 6): *«подтверждённый множитель ≈90x для USD-документов при курсе ≈90»* — канон конвертации был принят (`accepted`), но **не реализован в проде** (`cf-finance`), только в ad-hoc discovery-скриптах E1-T2/E1-T3.

**Слой дефекта:** `core.fact_payments` (Layer B, загрузка). `marts.expenses` (Layer C) корректно суммирует то, что ему дал `core` — дефекта на уровне марта нет. По закону-следствию главного принципа (`00_CHARTER`, ADR-002) — **фикс-форвардом правится Layer B (`cf-finance`), не `sq_marts_expenses`.**

## 8. Системность — ПОДТВЕРЖДЕНО (`run_e1t3_systemic_scope.log`)

Брифовский тест шага 7 (Σ opex mart vs core за весь май) отклонён как недиагностичный (см. выше) — заменён корректным тестом: live API (с верной конвертацией по ADR-010) vs `core.fact_payments`, по всем не-KGS документам мая, все статьи.

**Результат:** 65 не-KGS документов за май. 27 — легитимно исключены из ETL (`Перемещение исходящий`, `Закупка товаров` — `core_sum_kgs IS NULL`, подтверждает `02_ERP_CONTRACTS` "ИСКЛЮЧЕНО из ETL", не аномалия). Из оставшихся **38 документов, подлежащих загрузке — 38 из 38 (100%) показывают расхождение** между live API и `core.fact_payments`, во всех случаях в одну сторону (core занижает).

**Затронутые статьи (минимум, по факту в выборке):** Банк (комиссия), Мой Склад, Интернет и связь, Бухгалтерские услуги, Налоги и сборы, Расходы маркетплейсов, Вывод прибыли.

**Σ расхождений по 38 документам = 1 108 075,75 KGS** (из них 890 187,50 = доля «Вывод прибыли»; 217 888,25 — доля остальных ≥6 статей).

**Вывод (доказательно, с числами): механизм СИСТЕМНЫЙ, не ВП-специфичный.** Баг воспроизводится на 100% выборки не-KGS документов независимо от статьи расходов — это свойство самого шага загрузки `paymentout` в `core.fact_payments` (`cf-finance`), а не что-то специфичное для «Вывод прибыли». Прежние 0,00-совпадения E1-T2 (Банк (комиссия), Мой Склад и др. против оракула, `recon_expenses_sources_2026-05.md §1`) сравнивали **live-API-суммы** (корректно посчитанные) с оракулом — никогда не проверяли `core.fact_payments`/`marts.expenses`. Метод E1-T2 был слеп к этому багу; это не противоречие, а разный объект сверки.

**Важное следствие для миссии проекта (`00_CHARTER §миссия`):** `marts.expenses` — витрина для инвестиционной оценки — в текущем виде занижает минимум 7 категорий расходов одновременно, не только «Вывод прибыли». Полный масштаб (весь март, все месяцы, + `cashout`) этой сессией не измерен — это отдельный, более широкий вопрос, чем закрытие Q-27.
