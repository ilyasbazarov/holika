## Пункт 1 — Каноническое место хранения кода пайплайна (decision-половина Q-3, cf-finance-часть)

**Контекст:** `cf-finance` (revision `00006-piv`) извлечён и версионирован снапшотом в `/reference/code/cf-finance/` (см. `MANIFEST.md`, disk==deployed для кода подтверждено). Это read-only снапшот, не живой репозиторий с историей.

**Дополнительный факт, меняющий картину вариантов:** на самом Cloud Shell диске внутри `/home/ilyasbazarov4/cf-finance/` уже существует локальный каталог `.git` (обнаружен в `ls -la`, не исследовался глубже — содержимое/remote/история коммитов не читались, вне scope шагов брифа). Это означает, что вариант «(a) отдельный код-репо» может быть не «завести с нуля», а «запушить уже существующую локальную git-историю» — если она релевантна и не мусорная. Требует отдельного discovery (`git log`, `git remote -v` на диске) прежде чем на это опираться как на факт истории, а не как на голую возможность.

**Вопрос:** где канонически хранить исходники пайплайна — (a) отдельный код-репозиторий (возможно, посев из уже существующего локального `.git`), (b) в самом `holika`-репо (напр. `/pipeline-src/cf-finance/`), (c) снапшот-only в `/reference/code/` (как сейчас, без живой истории)?

**Варианты:**
- (a) Отдельный код-репо — чище разделение docs-репо / код-репо, штатно для типичного проекта; но означает ещё один репо для пиннинга по SHA в будущих брифах (усложняет `_METHOD §10` механику доступа).
- (b) Внутри `holika` — один репо для всего, проще пиннинг по единому SHA; но смешивает LIVING-доки метода с кодом, разная скорость изменений (код меняется при каждом деплое, доки — по сессиям).
- (c) Снапшот-only (текущее состояние этой сессии) — минимальный шаг, не решает проблему живой версионируемости на будущее, каждый новый деплой снова не отслеживается.

**Рекомендация разработчика (не решение):** (a) при условии, что локальный `.git` на диске окажется пригодным (не мусорным) после отдельной проверки — экономит работу по восстановлению истории. Если `.git` окажется пустым/нерелевантным — тогда (b) проще в рамках уже работающего docs-driven метода этого проекта (один SHA-пиннинг на всё). (c) — недостаточно как финальное решение, годится только как промежуточный шаг (уже сделан).

**Блокирует:** финализацию `NEW_DECISIONS` proposed-ADR этой сессии (черновик ниже, статус `proposed` до решения архитектора).

## Пункт 2 — `expand=rate` не задан в API-запросе cf-finance (потенциальный второй компонент фикса)

**Контекст:** см. `cf_finance_converter_findings_2026-07-14.md §1`. HTTP-запрос `cf-finance` к МойСклад строится с `expand=expenseItem,agent,project,salesChannel` — `rate` не входит. Неизвестно (не проверено в этой сессии), приходит ли поле `rate.value` в ответе МойСклад API без явного expand для `paymentout`/`cashout`.

**Вопрос:** нужно ли `E1-T3-MECH-FX` включать правку `expand`-параметра запроса (не только формулы `sum_kgs`), или `rate` уже доступен без expand и это чисто вопрос формулы?

**Рекомендация разработчика (не решение):** быстрая проверка одним `curl` к `entity/paymentout` с текущим `expand` (без правки кода) — покажет, есть ли `rate` в ответе уже сейчас. Дешёвый discovery-шаг, можно сделать как первый шаг `E1-T3-MECH-FX`, а не отдельным брифом.

**Блокирует:** точность оценки объёма правки в `E1-T3-MECH-FX` (не блокирует эту задачу, не блокирует принятие root-cause).
