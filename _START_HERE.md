# _START_HERE — Как запускать сессии с ролями

Одностраничный раннбук: что вставить в новый чат, чтобы модель приняла **правильную роль** и пошла
работать. Держи открытым рядом. Полный метод — `_METHOD.md`.

---

## 0. Три вещи, которые ты прикладываешь ВСЕГДА

1. **Файл роли** — вставляешь его целиком первым сообщением (или даёшь его raw-URL по SHA).
2. **Актуальный SHA** — короткая строка: `SHA сессии: <хэш>`. Модель обязана читать репо по нему.
3. **Задача/вопрос сессии** — конкретно, одним абзацем.

## ★★ Железное правило доступа (повтори модели, если забыла)

> Ходи в репо `ilyasbazarov/holika` **только через `curl` по приложенному SHA**. НЕ через
> web-fetch/браузер — он отдаёт кэш, а коммиты летят часто. Работать на устаревшем контенте запрещено.
> SHA не приложен → **спроси**, не читай `main`.

Пример чтения:
```
curl -s https://raw.githubusercontent.com/ilyasbazarov/holika/<SHA>/07_STATE.md
```

## 1. Карта репо (все рабочие доки — модель ходит сюда за фактами)

Подставь `<SHA>`. Прикладывай этот список в любой инициирующий чат, чтобы агент мог достать/сверить что угодно.

```
# метод и роли
_METHOD.md · 05_CONVENTIONS.md · _ARCHITECT.md · _APPLIER.md · _GENERATOR.md · 08_TASK_BRIEF_TEMPLATE.md · _START_HERE.md
# живые доки
04_ROADMAP.md · 06_DECISIONS_LOG.md · 07_STATE.md
# миграция и источник
TZ-MIGRATE-TO-DOCS-METHOD.md · PROJECT_REFERENCE_v6.md · RUNBOOK_v8.md
# STABLE/доменные (по мере P3–P5)
00_CHARTER.md · 01_ARCHITECTURE.md · 02_ERP_CONTRACTS.md · 03_*_SPEC.md · 09_GLOSSARY.md
# брифы задач
briefs/<TASK-ID>.md

base: https://raw.githubusercontent.com/ilyasbazarov/holika/<SHA>/<файл>
```

## 2. Запуск по ролям

### 🧠 Архитектор — решения, дизайн, гэпы, ревью
**Когда:** закрыть открытый вопрос, адъюдицировать ADR, разрулить гэп/противоречие, проверить, что шаг не
ломает границу контракта, поставить discovery-бриф.
**Вставляешь:** `_ARCHITECT.md` целиком → `SHA сессии: <хэш>` → вопрос.
**Модель сама** прочитает `_METHOD`+`00`+`04`+`06`+`07` (по SHA) и то, что нужно вопросу.
**На выходе:** session-блок (решения → `06`, состояние → `07`) + явные drop-in-патчи.

Launch-строка:
> Ты — архитектор проекта holika. Роль/правила — прикреплённый `_ARCHITECT.md` + `_METHOD` + `05_CONVENTIONS`.
> Читай репо `ilyasbazarov/holika` ТОЛЬКО curl по SHA: `<хэш>`. Обязательный контекст — `_METHOD`,`00`,`04`,`06`,`07`.
> Anti-improvisation: гэп → `CONTEXT GAP`/discovery-бриф, не догадка. Вопрос сессии: <…>.

### 🛠 Генератор брифов — собрать бриф следующей задачи
**Когда:** нужно превратить «текущий фокус» из `07` в исполняемый бриф.
**Вставляешь:** `_GENERATOR.md` целиком → `SHA сессии: <хэш>`.
**На выходе:** готовый `briefs/<TASK-ID>.md`. Коммитишь его, затем запускаешь разработчика.

Launch-строка:
> Ты — генератор брифов проекта holika (прикреплённый `_GENERATOR.md`). Читай репо ТОЛЬКО curl по SHA: `<хэш>`.
> Собери бриф задачи «текущий фокус» из `07_STATE`. Если задача с суждением и есть нерешённые ADR — сначала подними их.

### ⚙️ Разработчик — выполнить задачу по брифу
**Когда:** есть готовый бриф.
**Вставляешь:** содержимое `briefs/<TASK-ID>.md` целиком → `SHA сессии: <хэш>`. Бриф сам несёт «context-to-load».
**Модель:** ПИШЕТ код/артефакты + команду запуска; ты ЗАПУСКАЕШЬ и возвращаешь логи; она анализирует.
**На выходе:** артефакт + session-блок. «Done» — только после подтверждённого лога.

Launch-строка:
> Ты — разработчик проекта holika. Сначала прочитай (curl по SHA `<хэш>`) `_METHOD`+`05_CONVENTIONS`, затем бриф ниже.
> Ты пишешь, человек запускает. Anti-improvisation. В конце — session-блок. Бриф: <вставить `briefs/<TASK-ID>.md`>.

### 📥 Applier — применить session-блок и получить файлы для коммита
**Когда:** сессия закончилась, есть session-блок.
**Вставляешь:** `_APPLIER.md` целиком → session-блок → полные тексты затрагиваемых доков (`07` всегда;
`06` если `accepted` ADR; `05` если одобренные конвенции; доки discovery-обогащения если есть) → `SHA: <хэш>`.
**На выходе:** обновлённые ПОЛНЫЕ файлы. Копируешь и коммитишь.

## 3. Рабочий цикл (петля)

```
07 «текущий фокус»
      │
      ▼
[Генератор] ──► briefs/<TASK>.md ──► [Разработчик] ──► артефакт + session-блок
      ▲                                                          │
      │                                                          ▼
[Архитектор] ◄── гэп/решение/ADR ◄────────────── [Applier] ──► коммит (07/06/доки)
      │
      └──► discovery-бриф (при гэпе) ──► выгрузка/расследование ──► обогащение репо ──► коммит
```

После коммита — **новый SHA**; прикладывай его в следующую сессию.

## 4. Что делать прямо сейчас — по-человечески

Осталось два коммита, и можно запускать первую рабочую сессию:

1. **Залей в репо два своих исходных файла** — `PROJECT_REFERENCE_v6.md` и `RUNBOOK_v8.md`. После
   заливки не редактируй их: пока идёт переезд, это замороженный источник правды.
2. **Залей `08_TASK_BRIEF_TEMPLATE.md`** — его сейчас в репо нет.

Когда оба сделаны — открываешь новый чат и запускаешь **архитектора** (см. раздел 2, «Архитектор»).
Прикладываешь: файл `_ARCHITECT.md`, свежий SHA, оба исходных файла и файл `TZ-MIGRATE-TO-DOCS-METHOD.md`.
Пишешь простую задачу: «разбери оба исходных файла по кусочкам и составь карту переезда — что куда
переносим, что выкидываем, чего не хватает». На выходе он даёт файл `MIGRATION_MAP.md` — это твоя карта,
дальше двигаемся по ней.
