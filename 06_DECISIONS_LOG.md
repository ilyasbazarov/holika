## ADR-000 · Принят docs-driven multi-agent метод
- **Контекст:** проект вёлся двумя документами — PROJECT_REFERENCE_v6 и RUNBOOK_v8; не масштабируется,
  растит контекст чата, смешивает стабильное и живое, нет append-only истории решений и разделения ролей.
- **Решение:** перейти на метод из `_METHOD.md` — плоский набор нумерованных `.md`, роли
  архитектор/разработчик/applier/человек, агенты читают репо по SHA, знание фиксируется session-блоками.
- **Последствия:** заведён репозиторий `ilyasbazarov/holika`; каркас ролей (`_ARCHITECT`, `_APPLIER`,
  `_GENERATOR`, `05_CONVENTIONS`, `08`) и `_METHOD` — в репо. Миграция ведётся по `TZ-MIGRATE-TO-DOCS-METHOD.md`.
- **Статус:** accepted

## ADR-001 · Отдельный репозиторий
- **Контекст:** проект не связан с другими (в т.ч. с эталонным `ilyasbazarov/docs`, откуда взят метод).
- **Решение:** вести в отдельном репо `ilyasbazarov/holika`, со своим потоком SHA и независимыми ролями.
- **Последствия:** все raw-URL в доках указывают на `ilyasbazarov/holika`.
- **Статус:** accepted

## ADR-002 · Граница контракта = эталонные цифры ERP (референс-оракул)
- **Контекст:** методу нужна жёсткая граница/красная линия. У проекта она есть — эталонные цифры в ERP.
- **Решение:** граница контракта — **референс-оракул приёмки** (`_METHOD.md` §4b): выход проекта обязан
  биться с эталонными цифрами ERP «по цифрам»; расхождение закрывается правилом/override/ADR, не тихим
  примирением. Доступ к ERP есть; недостающие эталоны выгружаются руками в `/reference`.
- **Последствия:** `00_CHARTER` §главный принцип и `02_ERP_CONTRACTS` фиксируют, что за цифры и как
  выгружаются. Точный состав эталона — открытый вопрос, закрывается discovery-брифом в P3.
- **Статус:** accepted

## ADR-003 · Applier, генератор брифов и discovery-петля — с самого старта
- **Контекст:** источник заведомо неполон; двигаться по задачам без обогащения репо = гадать
  вслепую, жечь ресурс.
- **Решение:** задействовать полный набор ролей сразу, включая `_APPLIER` и `_GENERATOR`; discovery-петля
  (`_METHOD.md` §11) — центральный механизм, не опция. Гэп → discovery-бриф (return-this = обогащение
  репо), не догадка.
- **Последствия:** `07_STATE` держит GAP-реестр; генератор умеет discovery-бриф; applier коммитит
  обогащение (`/reference`, ADR, патчи доков).
- **Статус:** accepted

## ADR-004 · Ввести 11_INFRA_FACTS (LIVING)
- **Контекст:** волатильные инфра-факты (URL/ревизии CF, Config ID SQ, расписания, IAM, секреты) не помещаются в STABLE 01/02 без нарушения дисциплины изменений.

Карта доков `00_CHARTER` включает `11` в LIVING-слой.

## ADR-005 · Границы референс-оракула приёмки (seed #1 — П&Л май-2026)
- **Контекст:** ADR-002 задал границу = эталонные цифры ERP, но состав/гранулярность/правило сверки/закрытие расхождений (Q-1) не определены. PR-28 сам документирует известный разрыв ~33% между fact_payments и П&Л МойСклада — эталон и выход НЕ сходятся by design; риск принять выход за эталон.
- **Решение:**
  - (1) Ground-truth = сторона МойСклада (П&Л), НЕ fact_payments. В seed все числа PR-28 дословно, с разметкой класса ORACLE / PIPELINE_SNAPSHOT / DERIVED.
  - (2) Гранулярность = статья П&Л (line-level), не только итог.
  - (3) Сверка = точное совпадение (до копейки) на ORACLE; PIPELINE_SNAPSHOT не сверяется.
  - (4) Целевая сверка расходов: fact_payments (+ будущий entity/loss) → 11 132 303,89, с известным разрывом как зарегистрированным исключением Q-8/Q-9/Q-10; допуск = документированный остаток, пока Q-8 не закрыт; закрытие = discovery→ADR, не подгонка.
  - (5) RECON-цифры (выручка 93 251 530,80; «В пути» 104 877 706,26) — DEFER в `/reference/recon_2026-05.md`; точка возврата: свежая выгрузка МойСклада ИЛИ ADR, принимающий PR-строки 21/25/47 как провенанс.
- **Последствия:** создан `/reference/pnl_2026-05.md`; `02_ERP_CONTRACTS` §оракул регистрирует артефакт+правило; Q-1 → partial; приёмка «по цифрам» включается на ORACLE-строках после апрува.
- **Статус:** accepted

## ADR-006 · Методология агрегации расходной части П&Л МойСклада и карта категорий дашборда
- **Контекст:** Q-8 требовал установить, как строится расходная строка П&Л МойСклада. Обнаружено, что 22 из 23 статей строятся из трёх источников (paymentout + cashout + loss) с конвертацией по курсу валюты документа; единственная исключение — статья «Комиссия», которая строится отдельно из `entity/commissionreportin` (сумма reward÷100 + commissionOverhead.sum÷100 по документам периода). Все 22 обычные статьи подтверждены точным совпадением (0.00 разницы) с ручной выгрузкой П&Л 2026-07-06; методология Комиссии проверена на 7 документах мая с точным совпадением.
- **Решение:**
  - (1) Зафиксировать двухисточниковую методологию агрегации П&Л как основную для построения марта расходов: источник #1 = paymentout+cashout+loss (22 статей + FX конверсия по курсу документа), источник #2 = entity/commissionreportin (1 статья, формула reward÷100 + commissionOverhead.sum÷100).
  - (2) На дашборде категория «Расходы маркетплейсов» объединяет обе статьи — источник #1 (статья «Расходы маркетплейцов» из paymentout/cashout/loss) И источник #2 (entity/commissionreportin reward+overhead) — решение владельца от 2026-07-06.
  - (3) Статьи вне списка 27 целевых категорий дашборда (переданного заказчиком) исключаются как движения капитала/финансирования, не операционные расходы: Перемещение исходящий, Благотворительность, Возврат займа собственнику, Выплата тела кредита, Проценты по кредиту, Возврат.
  - (4) Нулевые значения по статьям (IT, Аренда, Фулфилмент в мае-2026) — ожидаемы (подтверждено эталоном 2026-07-06), не пробел в данных.
  - (5) ORACLE seed #1 (ADR-005, 2026-06-25) заменяется на seed #2 (эталон 2026-07-06, свежий ручной снимок из МойСклада), поскольку seed #1 содержал ошибки ввода (гипотеза дрейфа данных отклонена владельцем).
- **Последствия:**
  - `02_ERP_CONTRACTS` §семантика П&Л — заменить комментарий GAP Q-8 на эту двухисточниковую методологию с примерами агрегации и ссылкой на entity/commissionreportin.
  - `03_PIPELINE_SPEC` — новая секция §marts.expenses с логикой построения марта и картой 27 категорий дашборда (от заказчика).
  - `/reference/pnl_2026-05.md` — заменить эталон на свежий снимок 2026-07-06 с той же разметкой ORACLE/PIPELINE_SNAPSHOT/DERIVED.
  - `07_STATE` — Q-8 → closed; Q-9/Q-10 оставляют блокирующие зависимости для UUID резолва и кода, но методология от них независима (работает по имени статьи).
- **Статус:** accepted

## ADR-007 · Три порога свежести — не примиряются, документируются по механизму+scope
- **Контекст:** источник (PR-19/PR-31/RB) содержит три разных порога «свежести», выглядящих как противоречие: DQ 3 дня, алерт 6 часов, ручная проверка 2 часа. Наполнение `03 §DQ` (M-P4-03d) требует решения: свести в одно число или развести. Установлено, что пороги принадлежат трём независимым механизмам и частично меряют разные колонки (`transaction_date` vs `_loaded_at`).
- **Решение:**
  - (1) Пороги НЕ примиряются в одно число. `03 §DQ` документирует все три в ОДНОЙ таблице с колонками: порог · механизм · метрика(колонка)+scope · назначение · действие при нарушении.
  - (2) Классификация по механизму: 3 дня = `cf-dq`/DQ Gate (блокирующий, автомат) на `MAX(DATE(transaction_date_raw))`; 6 часов = `cf-alert`/Telegram (уведомляющий, автомат) на `MAX(transaction_date)`; 2 часа = ручная/runbook liveness-проверка (человек) на `_loaded_at`.
  - (3) 2-часовой порог фиксируется как liveness-heartbeat на `_loaded_at` (пайплайн часовой → нет свежей загрузки за 2ч = пропущенный прогон). Формулировка еженедельного чеклиста RB (2ч на `transaction_date`) помечается как неточная, канон — `_loaded_at`.
  - (4) Ревизия `cf-dq` (Q-6 / TD-DQ-REVISION) — волатильный факт, дом `11_INFRA_FACTS`; на значения порогов не влияет и этот ADR не гейтит.
- **Последствия:**
  - `03_PIPELINE_SPEC` §DQ — таблица свежести (3д/6ч/2ч) по схеме (1)–(3); разблокирует M-P4-03d.
  - `07_STATE` — снять заметку-противоречие «пороги разбросаны»; ADR-007 → accepted.
  - `04_ROADMAP` — ветка примирения `ADR-007 → 03d` закрыта; `03d` стартует.
- **Статус:** accepted

### Пороги свежести (ADR-007 — три механизма, не примиряются)

| Порог | Механизм | Метрика (колонка) + scope | Назначение | При нарушении |
|---|---|---|---|---|
| ≤ 3 дня | cf-dq / DQ Gate (автомат, блокирующий) | `MAX(DATE(transaction_date_raw))`, бизнес-дата | Устаревание бизнес-данных в CORE; терпит пустые дни | DQ FAILED → блок promote; пустой день → manual promote (§16) |
| ≤ 6 часов | cf-alert / Telegram (автомат, уведомляющий) | `MAX(transaction_date)`, бизнес-дата | Оперативное уведомление о разрыве потока в течение дня | Telegram-алерт → §4 |
| ≤ 2 часа | ручная / runbook (человек, liveness) | `_loaded_at`, техническая дата загрузки | Heartbeat: запускался ли workflow (часовой пайплайн) | Диагностика §4.2 / §7 (silent skip) |

## ADR-008 · Audit-снапшоты — инструментация + латентный SCD-актив
- **Контекст:** `sq_audit_dim_{products,counterparties,employees}_snapshot` (ежедневный INSERT-append в датасет `audit`) обнаружены без §audit в 01–03/09, без потребителя (LS/марты не читают). Историчность dim'ов неравномерна: `dim_counterparties` = SCD2 в ядре; `dim_products` = без SCD2 (перезапись MERGE); `dim_employees` = схема не документирована (Q-4), вероятно без SCD2.
- **Решение:**
  - (1) Дом не в доменной спеке: Config ID/расписание/стратегия → `11_INFRA_FACTS` §SQ (PR-21); схема датасета `audit` → `/reference` (Q-4); SQL → `/reference/sql/` (Q-5); узел `audit` → `01 §топология` (1–2 строки).
  - (2) НЕ заводить `§04_AUDIT_SNAPSHOTS` в `03` и секцию в `09`: нулевая доменная логика, нет требования/оракула приёмки.
  - (3) Зафиксировать латентную роль: для `dim_products`/`dim_employees` снапшот = единственный носитель истории (паттерн SCD2-via-snapshot). Промоушен в consumer-facing спеку — только по появлении аналитического требования (тогда — новый ADR).
  - (4) Операционный флаг: INSERT-append без retention → линейный рост; retention/партиционирование `audit` — в `10_OPS_PLAYBOOK`/`11` (не блокер).
- **Последствия:** `01 §топология` +строка про `audit`; `11 §SQ` — три audit-SQ; `07_STATE` — Q-17 закрыт как «инструментация, промоушен по требованию».
- **Статус:** accepted

## ADR-009 · Легаси fact_payments-март расходов вытесняется raw-мартом ADR-006
- **Контекст:** живой `sq_marts_expenses` (CREATE OR REPLACE, PR-236) читает `core.fact_payments` — построен из 2 из 4 источников (только paymentout+cashout), задокументированный ~33% оракул-разрыв (ADR-005 §4, PR 433–436). ADR-006 §marts.expenses задаёт TO-BE из `raw.moysklad_*` напрямую. `EXCLUDE_EXPENSE_IDS` на Python ретайрнут (Ghost fix 2026-06-18); нынешняя чистка `fact_payments` = BQ DELETE по UUID после MERGE (Q-10). ADR-006 исключает 6 статей по ИМЕНИ, независимо от Q-9/Q-10.
- **Решение:**
  - (1) Живой `fact_payments`-based `sq_marts_expenses` = LEGACY. TO-BE = ADR-006 raw-март. Epic-1 строит TO-BE и ретайрит легаси.
  - (2) M-P4-03e2: SQL легаси-марта переносится в `03 §marts` с явной AS-IS/legacy пометкой (источник `fact_payments`, ~33% разрыв) + кросс-ссылка на `§marts.expenses` (TO-BE). Не примирять молча.
  - (3) Фильтрация raw-марта — по имени статьи (ADR-006), развязана от Q-9/Q-10, валидирована оракулом (0.00 diff 2026-07-06). ЗАРЕГИСТРИРОВАН риск: имя хрупко к переименованию/дублю. Future-item: по закрытии Q-9/Q-10 захардить исключения на UUID (сверив с именами) — defense-in-depth, не блокер Epic-1.
- **Последствия:** `03 §marts` — AS-IS/TO-BE разметка легаси-марта; `03 §marts.expenses` — примечание про имя-vs-UUID и future-хардинг; `07_STATE` — Q-15 закрыт (ADR-009), Q-16 закрыт (имя-based старт + UUID-хардинг future, связан с Q-9/Q-10).
- **Статус:** accepted (**§1 (build-framing) partially superseded by ADR-012** — см. ниже)

## ADR-010 · Конвертация мультивалютных сумм (minor-units × rate.value)
- **Контекст:** обнаружен корень TD-RECON-01 — конвертация `÷100 без ×rate` давала ~90× недооценку сумм в USD. Источник: PR-33 п.6, RB-38 §26.
- **Решение:** канон конвертации — `minor_units ÷ 100 × rate` документа (`rate.value` документа, **если задан; иначе — текущий курс валюты**) до суммирования в KGS; `÷100 без ×rate` запрещён. **Скоуп: все входящие мультивалютные суммы (`price`/`sum` позиций, `sum`/`payedSum` в `invoiceout`, `sum` в `paymentout`/`cashout`) — глобальный inbound-канон. Исходящая конвертация (KGS→USD через `cf-fx`/`dim_fx_rates`) — вне ADR-010 (Q-19).** Фикс применён 2026-06-25.
- **Последствия:** `02_ERP_CONTRACTS` §валюты:266 → ссылка на ADR-010; `09_GLOSSARY` §конвертация валют → кросс-ссылка.
- **Статус:** accepted (адъюдикация M-P5-AP, 2026-07-08 — правка fallback-приоритета и скоупа против исходной формулировки `proposed`)

## ADR-011 · Фильтрация статей расходов — только DELETE-after-MERGE (Ghost Records)
- **Контекст:** источник: PR-27, PR-34 п.26.
- **Решение:** выгружаются все платежи без исключений на этапе выгрузки; системные статьи удаляются `DELETE` после `MERGE`. Инвариант: никогда `if/continue` на этапе выгрузки (Ghost Records). **Статус утверждения: канон-по-источнику (документированное целевое поведение + TO-BE-инвариант); конформность живого кода `if/continue`-free НЕ верифицирована — гейт Q-3 (код не версионирован).** Полный `EXCLUDE_EXPENSE_IDS` = Q-10.
- **Последствия:** `03_PIPELINE_SPEC` §fact_payments:344 → ссылка на ADR-011.
- **Статус:** accepted (адъюдикация M-P5-AP, 2026-07-08 — добавлен caveat о неверифицированной конформности живого кода)

## ADR-012 · Прод-март расходов = `sq_marts_expenses` (fact_payments-based); TO-BE raw-март не строится
- **Контекст:** требовалось установить, что реально питает страницу «Расходы» и совпадает ли это с ADR-006 (Q-18). Репо противоречиво: ADR-009 говорит, что Epic-1 строит TO-BE raw-март и ретайрит легаси; владелец говорит, что проект уже построен и переделывать заново не нужно. Разрешено фактом из `/reference/sql/sq_marts_expenses.sql` @ снапшот 2026-07-07 (SHA `805450a`, M-P4-D5) и `bq ls`: страница «Расходы» ← кастом-запрос LS `msklad_expenses` (тонкая обёртка, фильтр по `DATE(moment)`, без логики) ← таблица `msklad-bi-prod.marts.expenses` ← scheduled query `sq_marts_expenses` (Config `6a22a243-0000-20fd-a458-883d24f4cad4`), который читает только `core.fact_payments` = `paymentout+cashout` (2 из 4 источников, ADR-009 стр.106 / ADR-005 §4). Нет `loss`, нет `commissionreportin`. Расписание: ежедневный автоматический прогон (BQ DTS default «every 24 hours», якорь = creationTime 2026-06-05 11:10:25 UTC → nextRunTime +24ч, фактические прогоны ~11:10 UTC, WRITE_TRUNCATE). ⚠ Исходная формулировка «ручной/on-demand» (снапшот D5 читал пустой timeBasedSchedule без nextRunTime) — исправлена по живой bq show-перепроверке 2026-07-08 (провенанс: /reference/bq_transferconfig_sq_marts_expenses_2026-07-08.txt). Группировка по `expense_item_name` (по имени, `COALESCE(…, 'Не указана')`); `total_sum_usd = SUM(sum_kgs) / последний курс` (единый текущий курс на всю историю). ADR-006 двухисточниковая методология (paymentout+cashout+loss+commissionreportin) в проде НЕ реализована — валидирована лишь ручной выгрузкой 2026-07-06.
- **Решение:**

## ADR-016 · Root cause Q-27 = currency-conversion баг в core-load (cf-finance), СИСТЕМНЫЙ; форма и приоритет фикса
- **Контекст:** E1-T3 доказательно установил (recon_vyvod_pribyli_2026-05.md §2–4,§7–8): `cf-finance` при загрузке в `core.fact_payments` считает `sum_kgs = sum÷100` БЕЗ `×rate.value` для не-KGS документов — неисполнение уже принятого ADR-010 (чей scope глобален и включает `sum` в `paymentout`/`cashout`). Декомпозиция Δ(A↔B)=890 187,50 + Δ(B↔C)=0,00 = разрыв Q-27 без остатка. Системность: 38/38 (100%) подлежащих загрузке не-KGS документов мая занижены, ≥7 статей, Σ 1 108 075,75 KGS. Замеренный класс документов — `paymentout` (все 65 non-KGS мая); `cashout` в выборке не-KGS документов не дал.
- **Решение:**
  - §1 Root cause Q-27 = currency-conversion в Layer B (core-load, cf-finance), НЕ в `sq_marts_expenses`. По закону-следствию (`00_CHARTER` §главный принцип, ADR-002) — фикс-форвардом в Layer B. **accepted** (доказано фактом).
  - §2 Механизм СИСТЕМНЫЙ, не ВП-специфичный: ≥7 категорий `marts.expenses` занижены, Σ 1 108 075,75/май. Витрина для инвестора (`00_CHARTER` §миссия) в текущем виде занижает расходы по этим категориям — цифры НЕ предъявлять инвестору как финальные до `E1-T3-MECH-FX`. Системность закрыта ФАКТОМ (recon §8 + настоящий §2), отдельный Q-номер не заводится. **accepted**.
  - §3 Фикс-форвард = ENFORCEMENT ADR-010 в cf-finance: обязательное `minor_units ÷ 100 × rate.value` документа (если задан; иначе текущий курс) для всех не-KGS inbound-сумм на загрузке в core, по `paymentout` и `cashout` (охват cashout — из глобального scope ADR-010, forward-корректность, НЕ из замера E1-T3). Настоящий ADR НЕ переопределяет ADR-010, а фиксирует его неисполнение в проде и директиву на приведение в соответствие. **accepted** (директива).
  - §4 Форма: ОТДЕЛЬНАЯ задача `E1-T3-MECH-FX` (не пункт E1-T1-MECH). Основания: другой слой (код cf-finance vs SQL SQ), другой блокер-профиль (гейт Q-3 — код cf-finance не версионирован), другая oracle-валидация (re-diff core vs live API). Staging-first — ТРЕБОВАНИЕ. Гейт: discovery-бриф извлечения/версионирования cf-finance-кода (двигает Q-3) до написания фикса. **accepted**.
  - §5 Приоритет: `E1-T3-MECH-FX` ВЫШЕ E1-T1-MECH — по глубине слоя (core-load питает все марты) и радиусу (≥7 категорий; потенциально весь не-KGS во всех core.fact_*, если converter общий — не подтверждено, Q-3), не по величине суммы. НЕ stop-world: discovery cf-finance (Q-3) и E1-T3-MECH-FX идут своим треком; E1-T1-MECH диагностика/дизайн — параллельно. Порядок CUTOVER гейтится Q-31 (маршрутизация E1-T1-MECH через cf-finance → cutover после currency-fix; иначе независимо). **accepted**.
  - §6 Scope доказанного vs отложенного: доказано paymentout-занижение (май). Не измерено: cashout / другие месяцы / весь март / sales-сторона → Q-30 (DEFER, owner-gated). Находка поднимает приор Q-19 (÷100-без-×rate на sales-стороне — тот же класс) — рекомендован к приоритизации вместе с Q-30. **accepted**.

## ADR-018 · Нейминг файлов review-очереди: обязательный ординал `-N` (≥1); grandfather существующих; правка клаузы ADR-015
- **Контекст:** ADR-015 задал `/reference/architect_review_queue_<date>.md` (один файл/дата, без суффикса на несколько сессий/день). Коллизия 2026-07-14: старые доки (`ADR-015 §Последствия` «Первый экземпляр — …_2026-07-14.md»; `07_STATE` строка E1-T3) ссылаются на un-suffixed `…_2026-07-14.md`, которого физически НЕТ; реальные файлы — `…-2.md` (адъюдикация ADR-016) и `…-3.md` (ADR-017/RQ-2/RQ-3). RQ-4. `05 Часть II` код-стандарт — заглушка.
- **Решение:**
  - §1 Конвенция (forward): `/reference/architect_review_queue_<date>-<N>.md`, ординал `N` **обязателен**, старт с `1`, инкремент на сессию в пределах даты. Bare-date форма (`…_<date>.md`) запрещена. `N` = порядок сессии дня (самодокументирует порядок). **accepted.**
  - §2 Datetime-штамп отклонён: над `-N` не даёт выигрыша в сортировке (оба сортируются), дороже на ручную диктовку в SESSION_LOG, ошибкоёмче. **accepted (выбор `-N`).**
  - §3 Существующие `…-2.md`/`…-3.md` — **grandfather, НЕ ретро-переименовывать**: append-only ADR уже цитируют эти имена (ADR-016 из `-2`, ADR-017 из `-3`); переименование создаст висячие ссылки в неизменяемом `06`. **accepted.**
  - §4 Висячая ссылка: un-suffixed `…_2026-07-14.md` — имя, никогда не материализованное. `07_STATE` (LIVING) строка E1-T3 → указать на физический `…-2.md`. `ADR-015 §Последствия` (append-only) не редактируется — настоящий ADR помечает его клаузу нейминга `superseded-in-part`. **accepted.**
- **Последствия:**

  - `05 Часть III` §«Архитекторская review-очередь (ADR-015)» — добавить правило обязательного `-N` (SEMI-STABLE, через этот ADR).
  - `07_STATE`: RQ-4 → CLOSED (ADR-018); строка E1-T3 — исправить ссылку на `…-2.md`.
