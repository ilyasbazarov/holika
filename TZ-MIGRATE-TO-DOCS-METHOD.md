## 1. Цель (BLUF)

- **Суть.** Перевести проект «<PROJECT-OTHER>» с двух исходных документов (PROJECT_REFERENCE_v6 + RUNBOOK_v8) на docs-driven multi-agent метод из `ilyasbazarov/docs`: плоский набор нумерованных `.md`, роли на моделях разной стоимости, агенты ходят в доки за контекстом, передача знаний — через session-блоки.
- **Причина.** Источник не масштабируется: он растит контекст каждого чата, смешивает стабильное и живое, не даёт append-only историю решений и не разделяет роли/слои. Метод это решает и уже проверен на боевом проекте.
- **Действие.** Выполнить миграцию **фазами P0–P6** (§5), где каждая фаза — коммитируемый и проверяемый инкремент, источник остаётся источником истины до cutover, а каждый его кусок трассируется в целевой док или получает явный DROP/DEFER. Никакого big-bang.

## 4. Метод, который переносим — инварианты (обязательно воспроизвести)

Это **must-have** ядро. При миграции целевой проект обязан получить всё ниже, адаптированное под свой домен.

**4.1. Роли (модели как агенты).**
- **Архитектор** (дорогая модель) — проектирует, планирует, правит доки, принимает решения (ADR), держит целостность, пишет брифы. Прод-код не пишет.
- **Разработчик** (средняя модель) — берёт ОДИН бриф, пишет код/артефакты с ожидаемым выводом и инструкцией запуска. Сам не исполняет.
- **Человек** — исполнитель и владелец решений: запускает, возвращает логи, коммитит.
- **Applier** (дешёвая модель) — механически применяет session-блок к файлам репо. Без творчества.

**4.2. Таксономия доков (плоский репо, нумерованные `.md`).** Целевой набор (адаптировать имена под домен):
- `00_CHARTER` — конституция: миссия, **главный архитектурный принцип / граница контракта**, scope (in/out), критерии успеха, карта документов. STABLE.
- `01_ARCHITECTURE` — топология/последовательность. STABLE.
- `02_*_CONTRACTS` — схемы/контракты граничных артефактов. STABLE.
- `03_*_SPEC` — доменная логика ядра. SEMI-STABLE.
- `04_ROADMAP` — эпики → задачи с ID + приёмка. LIVING. Правило: детально расписан только текущий эпик; будущие — только заголовки (иначе нарушение anti-improvisation).
- `05_CONVENTIONS` — нейминг, код-стандарт, **правила агента + протокол сессий**. Самый важный для переносимости. SEMI-STABLE.
- `06_DECISIONS_LOG` — ADR, **append-only**. LIVING.
- `07_STATE` — текущий статус, открытые вопросы, контрольные цифры, «текущий фокус». LIVING (каждая сессия).
- `08_TASK_BRIEF_TEMPLATE` — шаблон пакета-передачи разработчику. STABLE.
- `09_GLOSSARY` — доменные термины. SEMI-STABLE.
- `10_*_PLAYBOOK` — грабли по типам ресурсов/операций. LIVING.
- Каркас ролей: `_ARCHITECT.md`, `_APPLIER.md`, `_GENERATOR.md`, `briefs/<TASK>.md`.

**4.3. Дисциплины (законы).**
- **Anti-improvisation** — главный закон. Нет нужного дока/значения/решения → `CONTEXT GAP` и стоп. Не выдумывать, не «дотягивать», не угадывать.
- **Работа на фактах** — поднять релевантный док до высказывания; где факт не подтверждён доком — это гэп, а не догадка.
- **Дисциплина изменений** — STABLE-доки меняются ТОЛЬКО через ADR; `06` append-only; `07` — каждую сессию через `STATE_PATCH`; blast radius правки перечисляется полностью.
- **Граница контракта — красная линия** — слой не лезет через границу чинить чужой слой (фикс-форвард в нужный слой).
- **Untrusted-вывод / не-идемпотентные операции** — грязный лог недостоверен → чистый перезапуск; обрыв не-идемпотентной операции → сначала read-only диагностика, не слепой retry.

**4.4. Ритуалы.**
- **Старт сессии:** загрузить обязательный контекст (`00`+`05`+`07`+бриф) → переформулировать задачу своими словами → перечислить входы → проверить полноту.
- **Конец сессии:** ОДИН session-блок (`SESSION_LOG` / `STATE_PATCH` / `NEW_DECISIONS` / `NEW_CONVENTIONS`), иначе знание потеряно.

**4.5. Механика доступа.**
- Чтение — raw-URL, пиннить по SHA. Запись обратно — не прямым коммитом модели, только через session-блок → человек/applier коммитит.
- «Архитектор — роль, не чат»: свежий чат становится архитектором, загрузив `00`+`04`+`06`+`07`. Старые чаты ретайрятся. Брифы генерятся repo-driven, не в одном вечном чате.

**Инвариант-минимум:** для ЛЮБОГО рабочего чата обязательный контекст = `00_CHARTER` + `05_CONVENTIONS` + `07_STATE` + конкретный бриф.

## 5. Пошаговый план миграции (P0–P6)

Каждая фаза = отдельная сессия / брифуемая задача, заканчивается session-блоком и коммитом. Не начинать фазу N+1 без зелёной приёмки фазы N.

### P0 — Каркас и заморозка (setup, без фактов проекта)
- **Действия:** по §6 (решено: отдельный репо). Создать целевой репо. Положить источник в него как `PROJECT_REFERENCE_v6.md` + `RUNBOOK_v8.md` c пометкой «ЗАМОРОЖЕН, источник истины до cutover, править запрещено». Портировать доменно-нейтральный каркас: `_ARCHITECT.md`, `_APPLIER.md`, `_GENERATOR.md`, `08_TASK_BRIEF_TEMPLATE.md`, и часть `05_CONVENTIONS` = «правила агента + протокол сессий» (без нейминга/код-стандарта — они появятся из фактов проекта позже).
- **Приёмка:** репо создан; источник заморожен; каркас ролей читается по raw-URL@SHA; в каркасе НЕТ ни одного факта о «<PROJECT-OTHER>».

### P1 — Дистилляция метода (ВЫПОЛНЕНО, вход) 
- **Статус:** `_METHOD.md` уже собран и дан как вход. Это целевой шаблон миграции.
- **Действия:** прочитать `_METHOD.md`; сверить его инварианты (§4) с эталоном (§3) на предмет пропусков; расхождение → `NEW_CONVENTIONS` proposed, не тихая правка. Отдельное внимание — §11 `_METHOD.md` (discovery-петля): она — рабочий инструмент фаз P2–P5, т.к. источник неполон.

### P2 — Инвентаризация и карта переноса (страховочная сеть)
- **Действия:** прочитать `PROJECT_REFERENCE_v6.md` + `RUNBOOK_v8.md`. Разбить на атомарные единицы (раздел/утверждение/значение). Выдать `MIGRATION_MAP.md` — таблицу: `источник (§/строка) → целевой док(§) | DROP(причина) | DEFER(куда/когда) | GAP(вопрос Ilyas)`. **Прозу пока не переписывать.**
- **Ориентир по источникам:** `PROJECT_REFERENCE_v6.md` (справочник) обычно питает `00_CHARTER` /
  `01_ARCHITECTURE` / `02_ERP_CONTRACTS` / `09_GLOSSARY`; `RUNBOOK_v8.md` (регламент) — операционную часть
  `05_CONVENTIONS` / `04_ROADMAP` / `10_*_PLAYBOOK`. Это подсказка, а не жёсткое правило — вердикт по факту.
- **Ожидание:** источник **заведомо неполон** — вердикт `GAP` будет частым и это норма, не сбой. Каждый `GAP` = кандидат в discovery-бриф (`_METHOD.md` §11), а не повод встать. Тип гэпа помечать: факт / решение / референс-артефакт (ERP-эталон).
- **Приёмка:** каждая единица источника имеет ровно один вердикт; список GAP/DROP/DEFER явный; матрица покрытия = 100% учтено; GAP-реестр перенесён в `07_STATE` «Открытые вопросы».

### P3 — Скелеты доков
- **Действия:** создать нумерованные целевые доки как **DRAFT-скелеты** с заголовками и заглушками-ссылками на строки источника (из карты P2). **`00_CHARTER` — первым.** Граница контракта **известна**: эталонные цифры ERP как референс-оракул (`_METHOD.md` §4b) — выход проекта обязан биться с ними по цифрам. Зафиксировать это в `00_CHARTER` (главный принцип) и в `02_*_CONTRACTS` (что именно за цифры, как выгружаются). Точный состав эталона неизвестен → discovery-бриф на выгрузку из ERP.
- **Приёмка:** карта документов из `00_CHARTER` соответствует созданным скелетам; каждый скелет ссылается на свои источники в источнике; ничего не заполнено «по памяти».

### P4 — Наполнение и верификация по одному доку
- **Действия:** заполнять каждый док из источника **по секциям**, отдельными сессиями. Контент — только из источника (или GAP). После заполнения STABLE-доки замораживаются; в карте P2 единица помечается «MIGRATED».
- **Приёмка каждого дока:** трассируемость к источнику (каждый абзац → строка источника); 0 внесённых фактов без источника; расхождения источника (противоречия сам-с-собой) вынесены как ADR-кандидаты, не «примирены» тихо.

### P5 — Процессный слой
- **Действия:** засеять `06_DECISIONS_LOG` ретро-ADR по решениям, уже неявно зашитым в источник (каждое — с контекстом/решением/последствиями). Засеять `07_STATE` — текущее состояние, открытые вопросы, контрольные цифры, «текущий фокус». Сгенерить первый боевой бриф через `_GENERATOR.md`.
- **Discovery-очередь:** GAP-реестр из P2 превратить в приоритизированный список discovery-брифов; первым — те, что блокируют «текущий фокус». Выгрузки ERP-эталона класть в `/reference` + фиксировать в `02_*_CONTRACTS`.
- **Приёмка:** `06` содержит только решения, извлекаемые из источника; `07` даёт свежему чату полный минимальный контекст + актуальный GAP-реестр; первый бриф самодостаточен (roles, context-to-load с литеральными URL, приёмка, return-this, session-блок).

### P6 — Cutover и e2e процесса
- **Действия:** проверить матрицу P2 = 100% (MIGRATED/DROP/DEFER, ни одного «висящего»). Переключить источник истины с исходных доков на нумерованные доки. Источник → архив-референс (пометка «superseded by docs @ <SHA>, только историческая справка»). Прогнать одну реальную задачу через полный цикл ролей (архитектор→бриф→разработчик→applier→коммит) как приёмку **самого процесса**.
- **Приёмка:** нет единицы источника без вердикта; свежий чат по `00`+`05`+`07`+бриф выполняет задачу без додумывания; цикл ролей отработал без ручного клея.

## 6. Решения (зафиксированы Ilyas)

1. **Репо:** отдельный, не связан с прочими проектами.
2. **Граница контракта:** есть — **эталонные цифры ERP** (референс-оракул, `_METHOD.md` §4b). Выход проекта бьётся с ними по цифрам. Доступ к ERP есть; недостающие эталоны выгружаются руками в `/reference`.
3. **Масштаб ролей:** **applier + генератор брифов нужны с самого начала.** Причина: источник неполон, много знания придётся добывать через discovery и обогащать репо — без обогащения движение по задачам вслепую и вгадку, что бесполезно и жжёт ресурс. Discovery-петля (`_METHOD.md` §11) — центральный механизм проекта, не опция.
4. **Нейминг/язык:** как в эталоне — проза RU, идентификаторы EN (переопределяемо в `05_CONVENTIONS`).

**Остаётся выяснить в ходе (не блокирует старт):** что в источнике — `DROP` по определению (устаревшее/разовое/секреты); точный состав ERP-эталона (закрывается первым discovery-брифом в P3).
